## プラン詳細（設計メモ）

### 思想の核
こすりちゃんの不変要素（音声/テロップ文字列/出現タイミング/注釈）は**絶対変えない**。
メモリ参照: `/Users/ca01224/.claude/projects/-Users-ca01224-Desktop---VOYAGE-/memory/feedback_kosuri_caption_rule.md`

Remotion移行は**レンダリング層の差し替えだけ**であり、検証変数を増やさない。
変わるのは画像・動かし方・見せ方のみという思想は維持。

### 既存Remotionスキーマ（Root.tsx L23-36 参照）
```ts
type Scene = {
  clip: string;          // 動画クリップパス
  text: string;          // テロップ文字列
  emphasis: boolean;     // 強調フラグ
  durationFrames: number;
};
type CompositionSpec = {
  fps: number; width: number; height: number;
  scenes: Scene[];
  narration?: string;
};
```

### こすりちゃん側のデータ（要確認）
- `caption_segments: [{start, end, text}]` 秒単位のテロップタイムライン
- パターンごと: `placement` / `size` / `style` / `emphasis_words` / `motion_type` / 画像群
- narration音声は元動画固定

### 変換ロジックの要点
- `caption_segments` の秒 → `durationFrames`（fps乗算）
- `emphasis_words` の単語レベル強調 → Remotion側 `<Telop>` 内で `<span style>` 的に部分装飾
- 画像＋motion_type（zoompan等）→ Remotion側の `<Img>` + interpolate で再実装が必要
- **セーフエリアY配置: hook_top=20%, kick_bottom=72%** を維持（Meta/TikTok両対応）

### 潰したい疑問（検証で回答する）
- **テロップ再現**: 現行ffmpeg版のPNGオーバーレイ品質を、Remotionネイティブテロップで同等以上にできるか
- **レンダ時間**: 10パターン量産でffmpeg比較。遅ければ量産用途で失格
- **JSONスキーマ**: AIが出しやすい形になっているか（LLMプロンプトから直接叩けるか）
