# Degraded Mode 報告義務（パク1.2.5輸入）

> 出典: `AIパクくん1.2.5/CLAUDE.md` 🔴 degraded mode 報告義務
> 目的: 本来の力を発揮できなかった時の「静かな劣化」を防ぐ

---

## 根本原則

> **本来の力を発揮できなかった場合、劣化版のパイプラインで完了した場合は、必ず以下を報告すること。**

**静かにdegradeして「できました」と報告するのは禁止。**
ユーザーが「何を失っているか」を常に認識できる状態にする。

---

## 報告必須項目

### 1. スキップされた工程の一覧

- RAG構築
- ベクトル品質ゲート
- Embedding検証
- Gemini検品
- N1選定フレームワーク
- 競合リサーチ（DPro未接続時）
- その他、本来実行されるべき工程

### 2. なぜスキップしたか

- API未設定（GEMINI_API_KEY / DPRO_API_KEY）
- 権限不足（MCP接続失敗）
- データ不足（RAGが未構築 / 鮮度切れ）
- ツール未インストール（ffprobe / numpy 等）
- 前提スキル未実行（research-park → concept-park → banner-park の順序違反）

### 3. 具体的な対策

**何を設定すれば本来の品質になるか** を明記。

例:
- 「DPro MCPが未接続のため、競合スコアリングなし。`.mcp.json` にAPIキー設定で有効化」
- 「GEMINI_API_KEY_2/3未設定のため、並列生成3並列→1並列で動作。`.env` に追加キー設定で3倍速」

### 4. 本来のパイプラインとの品質差

- 「フルモード: first-pass率80% / degradedモード: 推定50-60%」
- 「フルモード: 5軸評価 / degradedモード: 3軸評価」

---

## VOYAGEでの運用

### 全スキル共通の出力セクション

スキル完走時、以下のセクションを出力に含める:

```markdown
## Degraded Mode チェック

| 工程 | 状態 | 理由 | 対策 |
|---|---|---|---|
| RAG構築 | SKIP | データ未蓄積 | `/research-park` 先行実行 |
| Gemini検品 | SKIP | API key未設定 | `.env` に GEMINI_API_KEY 追加 |
| 5軸評価 | 3/5のみ | 権限不足 | — |

**本来品質との差**: 推定60-70%（5軸のうち3軸のみ稼働）
```

### 具体例

#### banner-park 完走時
```
Degraded Mode チェック:
- フルモード稼働
- 全ての品質ゲートPASS
- 想定品質100%
```

#### 逆にこうなるケース（明示する）
```
Degraded Mode チェック:
- DPro benchmark参照スキップ（benchmark.json鮮度30日超）
  → 対策: benchmark.json 再取得
- 仮説多様性スコア未計測（vector-utils未接続）
  → 対策: vector-utils.md 手順実行
- 想定品質: 60-70%（競合データなし、多様性検証なし）
```

### Slack通知への統合

VOYAGE Slack bot が完了通知を出す際、degraded 項目があれば **警告絵文字**（⚠️）を冒頭に付与:

```
[通知] ⚠️ banner-park 完了 (degraded mode)

詳細:
- RAG参照なし
- 3/5 ゲートのみPASS
- 推奨: /research-park を先行実行
```

---

## 禁止事項

1. **「完成しました」だけの報告**（degraded 情報を隠蔽）
2. **「問題ありません」だけの報告**（何を失ったかを明示しない）
3. **「API未設定」を理由にスキップしながら黙って進める**
4. **フルモード前提の品質指標を、degraded mode の成果に適用**

---

## VOYAGE既存 pak-philosophy.md との整合

VOYAGE「小さな嘘をつかない」原則と完全整合:

> 「演出のための嘘、都合の良い誇張、わざとらしい仕込み — 全部やらない。本物の体験だけを見せる。」

degraded 状態を隠すのは「小さな嘘」。パクの哲学に反する。
本ファイルはこれを **運用レベル** まで具体化したもの。
