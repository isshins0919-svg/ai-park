## 2. 設計原則

### 原則1: デュアルエンジン
生成エンジンは常に2以上。単一依存を避ける。
- 画像: Nano Banana Pro（`gemini-3-pro-image-preview`）
- 動画: Grok Imagine Video + Nano Banana
- 音声: Fish Audio TTS

### 原則2: データ忠実
直感より、ランキングデータに忠実にエンジン選定・仮説生成。
`dpro_benchmark.json` / ベクトル類似度 / 勝ちCR実績を一次情報とする。

### 原則3: 戦略翻訳
クリエイティブは「生成」ではなく戦略の「翻訳」。
strategy.json という設計図を各フォーマットに翻訳する。

### 原則4: カントクパターン（2026-03-29 確立）
指揮官（CKO / キャプテン / カントク）は一切実行しない。診断と指名だけ。
実装は専門クルーに委ねる。
- `/kiji-cko` = 記事LP部門のCKO
- `/movie-kantoku` = 動画部門のカントク
- `/amazon-captain` = Amazon部門のキャプテン

### 原則5: 判断階層（2026-04-18 パク1.2.5輸入）
CR評価は **愛 → 偉大 → 可能** の階層で行う。下位FAILで上位評価しない。
参照: `.claude/knowledge/aipark-imports/three-souls-judgment-hierarchy.md`

### 原則6: Vector-First（2026-04-18 パク1.2.5輸入）
仮説は座標として追跡・比較・更新・校正できる形で設計する。
テキストだけの仮説は補助メモ、主記録はベクトル。
参照: `.claude/knowledge/aipark-imports/vector-first-constitution.md`

## 3. 3層認知設計（N1選定フレーム）

- **潜在層**: 新認知 = 「原因の気づき」+「解決策の方向性」
- **準顕在層**: 新認知 = 「既存の限界」+「新しい判断基準」
- **顕在層**: 新認知 = 「根本的な違い」+「パラメータ書き換え」

strategy.json の `layers` フィールドで各層のペルソナ・メッセージを定義する。

## 4. クリエイティブの3つの魂（2軸）

### 制作時（構成要素視点）
- 魂1: **キービジュアルそのもの** — 広告を超えて作品レベル
- 魂2: **キラーキャッチコピー** — 1行で買わせるレベル
- 魂3: **別の仮説** — 量ではなく仮説の数

### 評価時（判断階層視点、パク1.2.5輸入）
- 愛（土台）— N1感情への寄り添い / 相手否定禁止 / 恐怖→希望の道
- 偉大（フィルター）— 時間耐久性 / コンセプト浸透率 / 非コモディティ
- 可能（拡張）— 未探索の可能性 / AGI活用度 / パターン超越

**両軸は補完関係**。制作で「作る」、評価で「評価する」。

## 5. フォーマット別コミュニケーション哲学

同じコンセプトでも、フォーマットによって「伝え方」が変わる。

### バナー — 1秒で新認知のフックを刺す（興味の入口）
- 伝えるもの: フック1本（新認知の断片）
- N1の脳内変化: 「え、何それ？」

### ショート動画広告 — 30秒で感情体験させる（Before→新認知→After）
- 伝えるもの: 新認知の感情体験
- N1の脳内変化: 「あ、自分のことだ…」→「え、そうだったの？」→「これ欲しい」

### 記事LP — 5分で論理的に納得させる（問題深掘り→新基準→商品証明→購買決定）
- 伝えるもの: 新認知の論証
- N1の脳内変化: 「なるほど、だからこれか」

## 6. エージェント設計の基本（パクさん直伝 + パク1.2.5輸入）

### 7原則
1. 目的・ゴール・行動指針・制約条件を必ず持たせる
2. 1エージェント1タスク
3. まず1体動かしてから組織を拡大
4. 新エージェントは上司エージェントにレビューさせる
5. 自走ルールを渡す（「3回自分で試してから聞く」等）
6. 制約は後入れ（最初は自由に → 後から薬機法等チェック）
7. エンベディングなしで良いクリエイティブは作れない（200本で十分）

### 新規作成時の5本質問（2026-04-18 輸入）
1. それ、本当にAgent/Skillか？既存の拡張で済まないか？
2. 最悪の失敗は何か？ガードは張られたか？
3. KPIは2週間で測れるか？
4. 何を捨てるか？Non-goals 明示
5. 誰がいつ動かすか？孤児になっていないか？

参照: `.claude/knowledge/aipark-imports/agent-requirements-5-questions.md`

## 7. コンテキストコスト設計（Tier 0-3）

Claude Code の性能はコンテキスト管理で決まる:

| Tier | 内容 | 重要度 |
|---|---|---|
| **Tier 0** | CLAUDE.md + rules/ — 全メッセージに入る | 1行でも減らす |
| **Tier 1** | Read（knowledge等）— セッション中残る | 必要時だけ読む |
| **Tier 2** | commands/ — /skill起動で親に読み込み | 重いスキルは注意 |
| **Tier 3** | agents/ — 独立コンテキスト、親にコスト0 | 並列可 |

**原則**: 重い専門タスクは agents/、ユーザー対話が必要なものは commands/

## 8. degraded mode 報告義務（2026-04-18 輸入）

本来の力を発揮できなかった場合、必ず以下を報告:
- スキップされた工程
- なぜスキップしたか
- 具体的対策
- 本来との品質差見積もり

静かにdegradeして「できました」は禁止。

参照: `.claude/knowledge/aipark-imports/degraded-mode-reporting.md`

## 9. サブエージェント制限（Claude Code仕様）

**Agent toolで起動したサブエージェントには、MCP・Bash権限が渡らない。**

- MCP呼び出し・Bash実行 → 必ずメイン会話で実行
- サブエージェントの用途 → テキスト推論・分析・レビュー（ファイル読み書きのみで完結するタスク）
