# FV自動カット検出の革命 — 考え方の転換記録

> 2026-04-14 作成
> 連結FV動画13本を誤差0.000秒で自動分割できるようになった過程の記録。

## 起きた革命を一言で

**「AIに考えさせる」から「物理現象を測る」に切り替えた瞬間、精度が桁違いに上がった。**

---

## 3つの考え方の転換

### 転換1: 「サンプリング」から「全数検査」へ

#### Before（v1: 失敗）
```
動画を0.1秒おきに「つまみ食い」→ AIに「これ切り替わった？」と聞く
→ 最大0.1秒のズレが構造的に発生（しかもAIの判断でさらにブレる）
```

#### After（v3: 成功）
```
全1,147フレームを1つ残らず検査 → 物理的なスコアで切り替わりを特定
→ フレーム精度（0.033秒）で完全一致
```

#### 抽象化した教訓
> **「精度が必要な仕事では、サンプリングしない。全数検査する。」**
>
> 人間は「全部見るのは大変だから間引こう」と考えがち。
> でもコンピュータにとって1,147フレームを見るのは5秒の仕事。
> 「間引く」という人間の発想をそのまま適用したのが最初の失敗。

#### 応用先
- 動画の品質チェック（全フレームのブレ・ノイズ検出）
- テロップの表示タイミング検出
- BGMのビート検出（全サンプルを見る）
- 記事LPのスクロール離脱ポイント検出（全ピクセル行の変化量）

---

### 転換2: 「AIの判断」から「物理量の測定」へ

#### Before（v1: 失敗）
```
フレーム画像をGeminiに送る → 「これFVの切り替わり？」と質問
→ Geminiが「テロップだけの変化」と「FV切り替わり」を区別できない
→ 過検出や未検出が発生
```

#### After（v3: 成功）
```
ffmpegのscene filter → 各フレームの「ピクセル変化量」をスコア化（0.0〜1.0）
→ 主観的な「判断」ではなく、客観的な「測定」
→ 数値だから閾値で制御可能、再現性100%
```

#### 抽象化した教訓
> **「AIに"判断"させる前に、まず"測定"できないか考える。」**
>
> AIの判断は確率的でブレる。同じ入力でも違う答えが返ることがある。
> 一方、物理量の測定は決定的。同じフレームは何度測っても同じスコア。
>
> AI判断が必要なのは「意味の解釈」が必要な場面だけ。
> 「画面が変わったか」は意味の問題じゃなく、ピクセルの物理変化の問題。
> **問題の性質を見極めてから、道具を選ぶ。**

#### 判断マトリクス
| 問題の性質 | 最適な道具 | 例 |
|---|---|---|
| **物理量で測れる** | ffmpeg/OpenCV/numpy（決定的） | カット検出、音量変化、色変化 |
| **意味の解釈が必要** | LLM/エンベディング（確率的） | 「笑顔の素材を選ぶ」「訴求が変わったか」 |
| **両方必要** | 測定で絞り込み → AIで判断 | 候補をスコアで絞り→Geminiで最終判定 |

#### 応用先
- BGMのサビ検出 → 波形の振幅・周波数（物理量）で検出可能。AIに「サビはどこ？」と聞く必要なし
- テロップの開始/終了 → OCR + フレーム差分（物理量）で検出可能
- 動画の「つまらないシーン」検出 → これはAIが必要（意味の判断）

---

### 転換3: 「閾値フィルタ」から「構造パターン認識」へ

#### Before（v2: 失敗）
```
scene score > 0.4 のフレームを全部「カット候補」にする
→ テロップ変化（score 0.2-0.3）がカット候補から漏れる
→ 閾値を下げるとノイズが大量に入る
→ 閾値調整の「いたちごっこ」
```

#### After（v3: 成功）
```
FV切り替わりは「2連続ピーク（ペア）」として現れるという構造を発見
→ ペアの後半 = 次のFV最初のフレーム = カットポイント
→ 閾値ではなく「パターン」で判定するから、score 0.2でも正しく拾える
```

#### なぜペアが生じるか（構造の理解）
```
  FV-A の最終フレーム     FV-B の最初のフレーム
       ↓                      ↓
  ... [A][A][A] | [B][B][B] ...
                ^
          ここで2つの急変が起きる:
          1. [A]→[B] の変化（前FV最終→次FV最初）← ピーク1
          2. [B]→[B] の安定化の直前のフレーム      ← ピーク2（の逆）

  実際には:
  ... [A安定][A最終][B最初][B安定] ...
              ↑ score高  ↑ score高
              = ペア（0.5-0.8秒間隔）
```

#### 抽象化した教訓
> **「データの中にある"構造"を見つけたら、閾値フィルタは不要になる。」**
>
> 閾値は「この値以上」という1次元の判定。データの構造（パターン）を使えば、
> 閾値に頼らずに正確な判定ができる。
>
> これはマーケティングでも同じ。
> - 閾値思考: 「CTR 1%以上の広告を残す」→ 0.99%の良い広告を殺す
> - 構造思考: 「CTRの時系列で上昇トレンドにあるものを残す」→ まだ1%未達でも伸びてるものを拾える

#### 応用先
- 動画のシーン構成分析（シーン間の「ペア」パターンで構成を自動認識）
- BGMのリズムパターン検出（ビートの「繰り返し構造」）
- 広告CRの勝ちパターン認識（ベクトル空間上の「クラスタ構造」）

## v1→v3 の進化を並べて見る

| | v1（失敗） | v3（成功） |
|---|---|---|
| **検査** | 0.1秒おきサンプリング | 全フレーム検査 |
| **判定** | Gemini AIに質問 | ffmpegの物理スコア |
| **フィルタ** | 固定閾値 | ペアパターン認識 |
| **精度** | ±0.5秒 | ±0.000秒 |
| **コスト** | Gemini API課金 | 0円 |
| **速度** | 2分（API待ち） | 5秒 |
| **再現性** | 確率的（毎回微妙に違う） | 決定的（何回やっても同じ） |

**全ての指標でv3が勝ってる。** しかもv1の方が「高度」に見える（AI使ってるから）。
これが今回一番大事な教訓:

> **「高度な道具を使うこと」と「正しい結果を出すこと」は全く別の話。**

## 今回のエンベディング・CCDDとの関係

CCDDの勉強会で「動画フレームのチャンク化 → ベクトル化」を学んだ。
最初はこれを使おうとした（v1）。

結果、エンベディングは「このケースでは不要」だった。

**ただし、エンベディングが不要だとわかったこと自体がCCDDの学びの成果。**
なぜなら、CCDDで学んだ判断基準があったから:

> 「セマンティック検索できるからLLMが重要。できないなら今までのPythonと同じ機能」

これを逆に読むと:
> 「Pythonで十分な問題にLLMを使うな」

FVカット検出は「画面が物理的に変わったか」の問題であり、
「意味が変わったか」の問題ではなかった。
だからffmpeg（Python/C）で十分だった。

**道具の選択を間違わないために、問題の性質を見極める力が一番大事。**
それがCCDDの本当の学び。

## 応用ロードマップ — 動画Pro編集への展開

### すぐ使える（物理量測定系）
| 用途 | 手法 | 今回のどの転換か |

| **FV品質の自動評価** | Gemini + ベクトル類似度 | 「刺さるか」は意味の判断 |
