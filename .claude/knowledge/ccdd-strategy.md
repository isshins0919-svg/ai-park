## 方針1: ベクトル化を「CR制作」から「全ナレッジ」に拡張

### 現状
- `vector-utils.md` で Gemini Embedding → コサイン類似度 → 品質ゲート
- 対象: hookVector, concept, competitor_message, N1需要のみ

### 目標
- **knowledge/ の全ファイル** をベクトル化し、セマンティック検索可能にする
- 「pak-philosophy.md の中で、今回の案件に最も関連する教えはどれか」を意味で引ける

### 実装ステップ
```
Step 1: knowledge/ 全ファイルをチャンク分割（見出し単位 = 意味単位）
Step 2: 各チャンクを Gemini Embedding でベクトル化
Step 3: ChromaDB にローカル保存（pip install chromadb）
Step 4: スキル起動時に「関連ナレッジ」をセマンティック検索で取得
```

### チャンク戦略
| ファイル種別 | チャンク単位 | サイズ目安 |
|---|---|---|
| pak-philosophy.md | 見出し（##）単位 | 中（150-400文字） |
| hook-db.md / cta-db.md | 1エントリ単位 | 小（30-150文字） |
| client情報 | セクション単位 | 中（150-400文字） |
| ブランドガイド | 段落単位 | 中（150-400文字） |

### 効果
- スキルが「必要なナレッジだけ」をCWに読み込める → CW節約
- 「このクライアントに似た過去案件」を意味で引ける
- knowledge/ が増えても検索精度が落ちない

## 方針2: エンベディングを「永続ベクトルDB」として蓄積

### 現状
- CR制作時にその場でベクトル化 → セッション終了で消える
- all_vectors.json はファイルベースで保存されているが、横断検索できない

### 目標
- **全クライアント × 全案件のベクトルデータ** を1つのDBに蓄積
- 「過去に刺さったフック角度」「競合の訴求パターン」を横断検索できる状態

### 実装ステップ
```
Step 1: scripts/vector_store.py を作成（ChromaDB ラッパー）
  - コレクション: knowledge / hooks / concepts / competitors / clients
  - メタデータ: client_slug, created_at, category, layer

Step 2: 各スキルの出力時にベクトルDBへ自動保存
  - Research Park → 競合メッセージ・N1需要をDB追加
  - Concept Park → コンセプト・フック角度をDB追加
  - Banner/ShortAd Park → 生成CRのベクトルをDB追加

Step 3: 「勝ちゾーン」「危険ゾーン」の概念を導入
  - 勝ちCR: CTR/CVR上位のCRベクトルを集積 → 重心計算
  - 新CR: 勝ちゾーンとの距離で「刺さりそう度」を事前予測
```

### 類似度の判断基準（CCDDより）
| 類似度 | 意味 | アクション |
|---|---|---|
| 0.8以上 | ほぼ同じ | 差別化が足りない。角度変更 |
| 0.7-0.8 | 同カテゴリ | 良い方向性。微調整で使える |
| 0.5-0.7 | 薄い関連 | 新しい切り口として検討価値あり |
| 0.5未満 | 無関係 | 別方向 |

## 方針3: ループを閉じる — 3つのフィードバックループ

### CCDDの核心
> エージェントは一発で終わるのではなく、自分で評価して、ダメならやり直すループ構造で動く

### VOYAGE号に足りないループ

#### ループA: スキル自動改善ループ（リフレクション）
```
スキル実行 → 結果の品質スコア記録 → 3回実行後に傾向分析
  → 「このスキルはStep3で品質が落ちる傾向」→ Step3を自動修正
```

**実装案:**
- 各スキルの実行結果を `logs/skill_metrics.json` に記録
- `/park-kaizen` 実行時にメトリクスを分析 → 改善提案 → 自動適用

#### ループB: CRパフォーマンスフィードバックループ
```
CR生成 → 配信 → DPro MCPで実績取得 → ベクトルDBに「勝ち/負け」タグ付与
  → 次回のCR生成時に「勝ちゾーン」を参照
```

**実装案:**
- DPro API でCR実績を定期取得（cost_difference, digg_rate等）
- 実績 × CRベクトル を紐付けて蓄積
- 勝ちCRの重心に近い方向でCR生成を誘導

#### ループC: コンテキスト品質監視ループ
```
セッション中のループ回数をカウント → 3回以上NGが続いたら警告
  → サブエージェントに委譲 or セッション分割を提案
```

**実装案:**
- CLAUDE.md に「ループ3回NG → 自動でエージェント分割を検討」ルール追加
- CW圧迫の兆候（指示忘れ、同じミス繰り返し）を自己検知

## リフレクション設計の具体指針（CCDDより）

### やってはいけない指示
❌「品質を振り返って改善して」（曖昧すぎる）

### 正しい指示パターン
✅「各ステップの出力を比較し、以下の異常があったら修正して」
- トーンの急変（前ステップと比較して感情スコアが±30%以上）
- 情報の欠落（必須項目が前ステップにあったのに消えている）
- 矛盾（前ステップの結論と逆の主張が出現）
- 修正時は異常が発生したステップのみ再実行（全体やり直しはしない）

## CCDDの学びで即適用できるTips

### スキル設計
- **Shot（具体例）のバリエーション** を全スキルに追加すべき
  - 数より多様性が重要（正常系 + 異常系 + 境界ケース）
- **スクリプト vs スキル** の判断: 考える必要がない定型処理 → スクリプト化（CW消費なし）

### インストラクション
- 5要素チェック: 役割 / 目的 / 背景 / 制約 / 参考情報
- 「言わなかったらClaudeが間違った方向に進むか」で要否を判断

### エージェント分割の判断
- ループ3回以上NG → エージェントを分ける
- CW圧迫の4症状: ハルシネーション / 指示忘れ / ループ悪化 / 精度低下

### 多層防御（セキュリティ）
- バリデーション → スクリプトでやる（LLM不使用 = 確実な防波堤）
- 人間を挟むのは「金額が大きい操作」だけ
- 権限は最小限
