# Handoff スキル — セッション引き継ぎ

> プランを立てたセッションで実行はしない。次のセッションへ引き継ぐ。

---

## いつ使うか

- プランモードでエージェント・スキル・システムの設計が完了したとき
- 作業が途中で、次のセッションに続きを持ち越したいとき
- 「ここまでやった、次はここから」を確実に引き継ぎたいとき

**なぜプランを立てたセッションで実行しないか:**
コンテクストウィンドウがプランで消費された状態で実行するとハルシネーションが起きやすい。プラン=思考力全集中、実行=フレッシュなセッションで、という分離が品質を上げる。

---

## 実行手順

### Step 1: セッションの状況を整理
以下を自分でまとめる:

```
【セッション引き継ぎ】
日付: YYYY-MM-DD
目的: このセッションで何をしようとしていたか
完了済み: 何が終わったか
残タスク: 次のセッションでやること（順番付き）
プラン詳細: 設計した内容の核心（次セッションで読めば即続けられるレベル）
注意点: 次セッションで気をつけること
```

### Step 2: 引き継ぎファイルを保存
`.claude/handoff/handoff_YYYY-MM-DD_HH.md` に保存する。

### Step 3: 次のセッション用のワンライナーを生成
次のセッションを開始したとき、このワンライナーをコピペするだけで続きから始められる1文を出力する。

---

## 次のセッションでの使い方

```
前回のhandoffを読んで、続きから始めて
```

または `.claude/handoff/` の最新ファイルを読んでからスタートする。

---

## 🚨 実装セッション開始時の鉄則（2026-04-24 追加）

**handoff経由で実装セッションを開始するとき、"既存骨格が動くか"の確認を最初のタスクに必ず入れる。**

### なぜ
前セッションから時間が空いている場合、「プラン時に前提にしていたコード」が:
- 書きかけで放置されていた（動作未確認）
- そもそも実データとスキーマが噛み合ってない
- 依存ライブラリ/設定ファイルが欠落してる

という状態がある。これを前提に設計を進めると、実装後半で致命的ミスマッチに気づき**プラン全体やり直し**になる。

### 具体的なチェック手順（実装の最初にやる）
1. **「動かせるなら動かす」**: ビルド/コンパイル/起動/サンプル実行など、数分で判明する動作確認を最初に入れる
   - Remotionなら `npx remotion compositions` でcomposition一覧が出るか
   - Pythonアプリなら `python -c "import target_module"` が通るか
   - Webアプリなら dev server が立つか
2. **スキーマ/シグネチャが実データと噛み合ってるか確認**: 既存コードの型定義・関数引数と、今やりたいことの入力が一致するか。ズレていればスキーマ再設計をプラン段階に差し戻す
3. **依存ファイル/設定ファイルの存在確認**: entry point、config、tsconfig、pyproject 等が揃ってるか

### 発端ケース
- **こすりちゃんRemotion移行MVP（2026-04-23→24）**
  - プランで「既存Remotion骨格を拡張する」前提で進めたが、骨格は2026-04-01に書かれたまま動作未確認だった
  - かつスキーマが「クリップ連結型」で、こすりちゃん実データ（単一画像+時間窓テロップ）と致命的ミスマッチ
  - → 結局Root.tsxは書き直し、entry/config/tsconfigも新規作成。プラン時にこの確認があれば設計が1段精密になっていた

## 出力フォーマット

```markdown
---
# 🤝 HANDOFF — YYYY-MM-DD

## このセッションのゴール
[何をしようとしていたか]

## 完了済み ✅
- [完了したこと]
- [完了したこと]

## 次セッションでやること 🚀
1. [最初にやること]
2. [次にやること]
3. [その次]

## プラン詳細（設計メモ）
[次のセッションで「なんだっけ」にならないための核心メモ]

## 注意点 ⚠️
- [次セッションで気をつけること]

## 次セッション用ワンライナー
> `.claude/handoff/handoff_YYYY-MM-DD_HH.md` を読んで、[タスク名]の続きを始めて。最初のタスクは「[最初のタスク]」から。
---
```
