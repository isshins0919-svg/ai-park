01

はじめに

なぜこの話をするのか — 完璧を求めず、育てるということ

AI活用の最大の壁は「最初から完璧を求める」こと

AIを導入しようとする人の多くが、最初から「完璧なシステム」を設計しようとします。要件定義に何週間もかけ、アーキテクチャを完璧に描き、全部のケースを想定して......結局、何も動かないまま終わる。

実際は違います。何度も失敗して、壁打ちして、改善して、ようやくまともになる。僕が作った「記事フィードバックAI」もまさにそうでした。

この教科書は、僕が実際にたどった道をそのまま見せるものです。きれいに整理された「成功事例」ではなく、泥臭い試行錯誤の記録。失敗も含めて、すべてを公開します。

AIチームは「設計」するものではなく「育成」するもの。人間のチームと同じです。
02

原点

Phase 0 — そもそも何がしたかったのか（2026年3月初旬）

課題: 記事LPの品質チェックに時間がかかりすぎる

記事LP（ランディングページ）は、広告の成果を左右する最重要クリエイティブです。でも、その品質チェックには大きな問題がありました。

🕐
レビューに1〜3日

上長に提出してからフィードバックが返ってくるまで、最短でも1日。忙しいと3日待ち。その間、修正も配信もできない。

👤
基準が属人的

レビューする人によって言うことが違う。Aさんは「フックが弱い」、Bさんは「CTAを増やせ」。どっちが正しいの？

🔄
修正サイクルが遅い

修正→再レビュー→再修正...のサイクルが1週間以上かかることもザラ。その間に市場は動いている。

理想の姿

「記事のURLをポンと渡すだけで、プロ級のフィードバックが5分で返ってくる」—— これが僕が目指した世界でした。

Before: 手動レビュー
  • フィードバックまで1〜3日
  • 基準が属人的でバラバラ
  • 業務時間内しか対応できない
  • 修正サイクル: 1週間〜
VS
After: AI自動フィードバック
  • フィードバックまで約5分
  • 統一されたマーケティング基準
  • 24時間365日対応
  • 修正サイクル: 同日中
Phase 0の学び 最初にやったことは「何が課題か」を明確にすること。ツールより先に、課題の解像度を上げた。
「AIを使いたい」ではなく「この業務のこの部分が遅い/属人的」という具体的な課題定義が出発点。
03

最初の一歩

Phase 1 — 「とりあえず聞いてみた」（2026年3月9日〜）

最初は超シンプル: Claudeに直接聞いてみた

まず手を動かしました。Claudeのチャット画面にURLを貼って、「この記事にフィードバックください」と書いただけ。設計書なんて書いていません。

Phase 1: 最初のプロンプト
PROMPT
この記事LPを分析して、改善点を教えてください。
URL: https://example.com/article-lp
AI RESPONSE（イメージ）
この記事LPには以下の改善点があります。
・見出しがやや長いので短くすると読みやすくなります
・画像をもう少し大きくするとインパクトが出ます
・CTAボタンの色を目立つ色に変更してみてはいかがでしょうか
・全体的に良い構成ですが、もう少し具体例があるとさらに良くなります

結果は......悪くはない。でも、プロのマーケターの視点とは程遠い。「もう少し具体例があるといい」って、それは僕でもわかる。

この時点の問題点

💬
当たり障りない

「いいですね」「改善の余地があります」— 具体性がゼロ。何をどう直せばいいのかわからない。表面的なレビューしかできていない。

📉
マーケ視点が弱い

フック・CTA・信頼構造という「マーケティングのプロが見るポイント」を知らない。見た目の改善提案に終始している。

🔁
毎回同じパターン

健康食品の記事でも美容の記事でも、同じフォーマットで同じようなフィードバック。記事の特性に応じた評価ができていない。

Phase 1の学び AIは「何を評価すべきか」を教えないと、表面的なレビューしかできない。
人間に「この記事レビューして」と丸投げしても良いフィードバックが返ってこないのと同じ。「何の観点で」「何を基準に」を指定する必要がある。

哲学を教える

Phase 2 — マーケ哲学を教え込む（2026年3月23日）

転機: 師匠のマーケティング哲学を教え込んだ

転機は、師匠から学んだマーケティング哲学をClaudeに「ナレッジ」として丸ごと教え込んだことでした。

🎯
N1を見つける

「最も典型的な1人の顧客」を徹底的に理解する。ペルソナではなく、実在する1人の人間像。

🔗
コンセプト一貫性

記事の冒頭から最後まで、1つのコンセプトで貫く。ブレた瞬間に読者は離脱する。

🧠
脳内1番乗り

カテゴリーの中で最初に思い出される存在になる。それがマーケティングのゴール。

Projects機能を使って、記事LP哲学やフックパターンのDBをナレッジとして蓄積。Claudeの「思考の土台」を作りました。

Phase 2: 哲学を教えた後のプロンプト
PROMPT
あなたはマーケティング専門家です。
N1視点で、この記事のフック・信頼構造・CTAを評価してください。
評価基準:
- フック: 冒頭200文字で「自分のことだ」と思わせているか
- 信頼構造: 権威・数値・口コミの3層が揃っているか
- CTA: 購入への「必然性」が構築されているか
AI RESPONSE（イメージ）
【フック評価: 65/100】
冒頭で「40代の肌悩み」に触れているものの、N1が抱える具体的なシーンが描けていません。「朝、鏡を見た時のがっかり感」のような、感情に刺さる描写が必要です。

【信頼構造: 40/100】
口コミは3件掲載されていますが、権威性（専門家推薦/受賞歴）が欠落しています。証拠の3層構造のうち1層しか機能していません......

格段に良くなりました。「65/100」「3層構造のうち1層しか機能していない」— 具体的で、行動に移せるフィードバック。

でも、まだ「1つのAIに全部やらせている」ため、各分析が浅い。フックの分析をしながらCTAも見て信頼構造も評価して......人間で言えば、1人で6つの部署の仕事をやっているようなもの。

Phase 2の学び AIに「知識」を与えるだけで、回答の質が劇的に変わる。
最初に教えるべきは「使い方」じゃなく「判断基準」。何が良くて何が悪いのか、その哲学を共有することが最も効果的。

人間でも、1人で営業・マーケ・法務・デザインを全部こなすのは無理がある。AIも同じだった。

チームを組む

Phase 4 — マルチエージェント化（2026年4月1日）

最大の転機: 専門家チームへの分割

ここが最大のブレイクスルーでした。1つのAIに全部やらせるのをやめて、「専門家チーム」に分割したのです。

着想は人間の組織と同じ。会社には営業部・マーケ部・法務部がある。なぜ？1人で全部やるより、専門家がそれぞれの領域に集中した方が、圧倒的に質が高いから。AIも同じだと気づきました。

Before: 1つのClaude

全部1人でやる → 分析が浅い

  • フックも見る
  • CTAも見る
  • 信頼構造も見る
  • オファーも見る
  • 全部「まあまあ」の分析
VS
After: 6つの専門エージェント

各自が1つの視点を深掘り → 深い分析

  • フック君はフックだけ
  • CTA君はCTAだけ
  • 信頼君は信頼構造だけ
  • 各領域で圧倒的な深さ

最初に作った6体のエージェント

🎣
フック君

冒頭200文字のフック強度を専門評価。「読者が最初の3秒で離脱しないか」を判定。

📖
アーク君

物語の感情アーク（起伏）を分析。「読者の感情が正しく動いているか」を追跡。

🛡️
信頼君

権威・証拠・口コミの3層構造を評価。「この記事は信頼できるか」を検証。

🔘
CTA君

購入ボタンの配置と心理的必然性を分析。「ここで買いたくなるか」を判定。

💰
オファー君

価格設計と緊急性の演出を評価。「今すぐ買わないと損だ」と感じさせられているか。

⚔️
競合君

同業界の勝ちパターンとの比較分析。「このジャンルで勝てる記事か」を判定。

4つの重要な設計判断

  1. 1エージェント = 1次元
    フック君はフックだけ評価。CTAには言及しない。
    なぜ？→ 越境すると分析が浅くなる。人間でも「あれもこれも」頼まれると、全部中途半端になる。
  2. 並列実行
    6体のエージェントが同時に動く。
    なぜ？→ 6体が直列（順番）だと6倍遅い。並列（同時）なら、最も遅いエージェントの時間だけで済む。
  3. 数値スコアリング
    感覚的な「良い/悪い」ではなく、数値スコアで統一。
    なぜ？→ 改善の前後比較が可能になる。「フック65点→82点」は明確な進歩。「ちょっと良くなった」は曖昧。
  4. GO / REVISE / BLOCK の3段階判定
    「まあまあ」を許さない。配信OK / 修正必要 / 配信停止 の3択。
    なぜ？→ 曖昧な判定は行動につながらない。「もう少し改善するといいかも」では誰も動かない。
精度が爆上がりした瞬間 6つの専門家が同時に「自分の領域だけ」を深掘りすることで、1体の時とは比較にならない深さのフィードバックが出るようになった。
フック君が200文字を徹底分析している間に、CTA君は全CTAボタンの配置を検証し、信頼君は証拠の3層構造を数えている。この「同時並行の専門分析」が、マルチエージェントの真の力。

磨き続ける

Phase 5 — 壁打ちと改善の連続（2026年4月1日〜2日）

作って終わりではない。ここからが本当の話。

マルチエージェント化で精度は大幅に上がりました。でも、使ってみると次々と課題が見つかる。何度も何度も改善しました。

改善①: 命名問題

命名の失敗と学び
CMO → CKO への改名

最初、司令塔を「CMO（Chief Marketing Officer）」と名付けました。でもこれは「会社全体のマーケティング最高責任者」の意味。記事LP専門のリーダーにこの名前は大げさで、役割が曖昧になります。

→ CKO（Chief Kiji Officer）に改名。「記事（Kiji）」の専門家であることを名前で明確にしました。名前を変えただけで、AIの振る舞いが変わった。「自分は記事LPの専門家だ」というアイデンティティが明確になったからです。

学び: 名前は重要 エージェントの「アイデンティティ」が明確でないと、振る舞いがブレる。
人間も同じ。「なんでも屋」より「フック専門家」の方が、自分の仕事に集中できる。

改善②: CKO前処理の追加

v1.0: CKOは結果を受け取るだけ

各エージェントの結果を統合するだけの「受付係」。記事の文脈を理解していないまま、スコアを合算していた。

  • 全記事に同じ基準を適用
  • ターゲットの違いを無視
  • 均一的で画一的な評価
VS
v2.0: CKOが先に記事を読んで指示

まずCKOが記事全体を読み、N1プロフィールを分析し、各エージェントへの指示（agent_directives）を生成してから分析開始。

  • 「60代健康志向女性の記事」と認識
  • 各エージェントに文脈付きの指示
  • ターゲット特化の深い評価
前処理の効果 各エージェントの分析精度が劇的に向上した。なぜ？
「60代健康志向女性の記事」と「20代美容記事」では、評価基準が違う。CKOがこの文脈を設定することで、各エージェントが的確に動ける。
会社で言えば、「上司がプロジェクトの背景と目的を説明してからタスクを振る」のと同じ。

改善③: 外部AIの導入

Claude単体では限界がありました。特に画像分析と競合データ。そこで、外部AIを「得意なこと」だけ任せる形で統合しました。

改善④: 重み付きスコアの設計

最初は全エージェントのスコアを均等に平均していました。でも、ターゲットの「認知段階」によって、重要なポイントは異なります。

潜在層には「気づき」を与えるフックが最重要。顕在層には「今買う理由」を与えるオファーが最重要。これを数値化した。

改善⑤: Hard Block ルール

ある日、スコア92点の記事が配信されました。しかし、その記事には薬機法に抵触する可能性のある表現が含まれていた。スコアが高くても、法律違反があれば配信してはいけない。

Hard Block: 法律違反だけは点数で補えない 薬機法に違反する表現があった場合、たとえスコアが95点でもBLOCK（配信停止）にするルールを追加。
これはシステム設計の根幹。最初から入れておくべきだった。
教訓: セーフティネットは最初に設計する。「あとで入れよう」ではリスクが大きすぎる。

次のステージ

Phase 6 — 診断から治療へ（計画中）

「問題を指摘するだけ」じゃ足りない

現在のシステムは「診断」はできます。「ここが悪い、ここを直せ」と教えてくれる。でも、「じゃあどう直すの？」はまだ人間の仕事。これが次の壁です。

病院に例えると、今は「検査結果: 血糖値が高いです」と言ってくるだけ。本当に必要なのは「この薬を飲んで、食事をこう変えて、1ヶ月後に再検査しましょう」まで。診断→治療→経過観察の一気通貫。

09

教訓

この旅から得た7つの学び

7つの教訓

1
最初から完璧を目指さない

まず動くものを作って、そこから改善する。Phase 1は「URLをコピペして聞いただけ」。それでいい。動かさないと課題が見えない。

2
AIには「判断基準」を先に教える

使い方より先に、何を大事にするかを教える。マーケ哲学を教えた瞬間、回答の質が劇的に変わった。

3
1エージェント = 1専門性

全部やらせると全部浅くなる。フック君はフックだけ。CTA君はCTAだけ。専門特化が深い分析を生む。

4
名前と役割を明確にする
