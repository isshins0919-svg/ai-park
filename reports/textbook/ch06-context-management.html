WHAT IS CW

コンテキストウィンドウ（CW）= AIの「今の頭の中」

CWは「AIが一度に処理できる情報量の上限」。人間の短期記憶（ワーキングメモリ）と同じです。
Island 01で軽く触れた「7番目のパーツ」を、ここで深掘りします。

📚 机の上の例

机の上に書類を3枚広げて作業する → 集中できる、全部見える
机の上に書類を50枚広げて作業する → 大事な書類が埋もれる、間違った書類を見て判断する

CW = 机の広さ。広げすぎると作業効率が落ちる。

🎒 通学カバンの例

全教科の教科書を毎日持っていく → 重い、使わない教科書ばかり
時間割を見て必要な教科書だけ持っていく → 軽い、すぐ出せる

「必要なものだけ持つ」がCW管理の本質。

WHEN CW OVERFLOWS

CWが溢れると起きる4つの症状

🚨 CW圧迫の4症状

これが出たら「頭がパンク」しているサイン

症状何が起きているか人間で例えると
① ハルシネーション答えを知らないのに、もっともらしい嘘を生成する徹夜明けで記憶が曖昧なのに「たしかそうでした」
② 指示忘れ最初に伝えた指示が途中から無視される10個の頼み事をされて、後半の3個を忘れる
③ ループ悪化同じミスを何度も繰り返す疲れてるときにタイプミスを直しても同じ場所で間違える
④ 精度低下全体的にアウトプットの質が落ちるマルチタスクのしすぎで全部の仕事が雑になる
なぜこうなるのか

CWには指示、会話履歴、読んだファイル、生成中のテキスト…が全部入る。容量が埋まるほど「優先度の低い情報」に注意が行き届かなくなる。

TIER DESIGN

CWの4階層管理 — Tier 0 〜 3

CWを効率よく使うために、情報を4つの階層（Tier）に分類します。
これがClaude Code設計の最も重要な概念です。

CW管理の4階層

Tier 0: CLAUDE.md + rules/
 → 全メッセージに必ず入る。1行でも減らしたい
 → 例: セキュリティルール、個人情報保護ルール

Tier 1: Read（knowledge/ 等）
 → 読んだらセッション中ずっと残る。必要な時だけ読む
 → 例: マーケ哲学、フックDB、クライアント情報

Tier 2: commands/（Skills）
 → /skill で起動すると親に読み込まれる。重いスキルは注意
 → 例: /banner-park, /kiji-cko

Tier 3: agents/（サブエージェント）
 → 独立空間で動く。親のCWを消費しない
 → 例: kiji-hook-crew, movie-arc-crew

💡 重い処理はTier 3に任せる = 親のCWを守る
🏢 オフィスの例

Tier 0 = 会社の全社ルール（全員が常に従う）
Tier 1 = 部署の資料棚（必要な時だけ取り出す）
Tier 2 = 会議に持ち込む資料（持ち込みすぎると机が溢れる）
Tier 3 = 別室で作業する専門チーム（本部の会議室を使わない）

🎒 遠足の例

Tier 0 = お弁当・水筒（絶対持っていく）
Tier 1 = 地図・しおり（目的地で見る）
Tier 2 = おやつ（持っていくけど重い）
Tier 3 = 大型荷物はバスのトランクに（自分のリュックに入れない）

CLAUDE.MD

CLAUDE.md — AIの設計図（Tier 0）

CLAUDE.mdはClaude Codeが毎回最初に読むファイルです。AIの行動規範・判断基準・ファイル構造を定義します。
Tier 0なので、短くすることが最重要。

📄 CLAUDE.mdの構成要素

4つのセクションで必要十分

1

起動アクション

起動時に実行すること（identity.md読み込み、時刻取得、git pullなど）

2

コンテキストコスト設計

Tier 0-3の定義と、agents/ vs commands/ の判断基準

3

ファイル構造

commands/ / agents/ / knowledge/ / rules/ の役割と配置

4

技術設定

出力先ディレクトリ、環境変数（値は書かない！）、同期コマンド

テンプレート

starter-kit に CLAUDE-template.md があります。ミッションで使います。

PLAN MODE

なぜ「プランモードで考え、次のセッションで実行する」のか

これもCWの話です。プランニングと実行を同じセッションでやると、CWが溢れます。

CWの消費イメージ

【プランニング】 要件整理、設計、シミュレーション → CWの60%を消費

【実行】 コード書き、ファイル作成、テスト → CWの50%が必要

【両方同じセッション】 60% + 50% = 110% → CWオーバーフロー

【分離する】
 セッション1: プラン（60%使用）→ ハンドオフ
 セッション2: 実行（50%使用）→ フレッシュなCWで高品質
🏗️ 建築の例

設計士が設計図を描きながら同時にレンガを積む人はいない。
設計は設計、施工は施工。分けるから両方の品質が高い。

📝 試験勉強の例

勉強計画を立てる日と、実際に勉強する日を分けた方が効率がいい。
ハンドオフ = 勉強計画をノートに書いて、翌日の自分に渡す。

SUMMARY

Island 06 まとめ

🧠
3つの持ち帰り
① CWは有限。詰め込みすぎるとAIは壊れる（嘘、忘却、精度低下）
② 情報をTier 0〜3に分類して、必要なものだけCWに入れる
③ CLAUDE.mdはTier 0。短く。プランと実行はセッションを分ける

🏁 チェックポイント — この島で覚えること

  • CW圧迫の4症状: ハルシネーション・指示忘れ・ループ・精度低下
  • Tier 0-3の階層管理。重い処理はTier 3（agents/）に任せる
  • CLAUDE.mdは毎回読まれるTier 0。1行でも削る
  • プランと実行はセッションを分ける。ハンドオフで引き継ぐ

📖 この島の航海用語

コンテキストウィンドウ（CW）

AIが一度に処理できる情報量の上限。人間のワーキングメモリに相当。

Tier 0

CLAUDE.mdとrules/。全メッセージに必ず入る最優先層。短くすることが命。

Tier 1

knowledge/等のRead対象。読んだらセッション中残る。必要時だけ読む。

CLAUDE.md

Claude Codeが毎回最初に読む設計図。AIの行動規範・判断基準を定義。

ハンドオフ

セッション間の引き継ぎ。プランの要点を次セッションに渡す仕組み。

プランモード

設計だけに集中するモード。実行は別セッションで行う（CW溢れ防止）。

🧠 自分のCLAUDE.mdを書こう

starter-kit の CLAUDE-template.md をベースに、自分の業務用CLAUDE.mdを設計するハンズオン。

ゴール

自分の業務向けCLAUDE.md（Tier 0設計）を完成させる

  1. starter-kit/CLAUDE-template.md をClaude Codeに読ませる
  2. 以下の情報を入力:
    CLAUDE-template.md を読んで、以下の情報で
    私用のCLAUDE.mdを作成してください。

    # 私の設定
    - 起動時にやりたいこと: [例: 時刻表示、git pull]
    - 終了時にやりたいこと: [例: git push、ログ保存]
    - よく使うナレッジ: [例: 顧客リスト、ブランドガイド]
    - ルール: [例: 個人情報保護、承認フロー]
    - 出力先: [例: reports/ フォルダ]

    # 注意
    Tier 0なので、できるだけ短くしてください。
    200行以内が理想です。
  3. 出力されたCLAUDE.mdを読んで、不要な行を削る（短さが正義）
  4. Island 04-05で設計したSkillとエージェント組織図をファイル構造セクションに反映する
所要時間: 約25分
