KNOWLEDGE VS SKILL

Skill設計の前に — ナレッジとスキルの違い

この島に入る前に、初心者が100%ハマる混乱を先に潰しておきます。

❓ よくある混乱

「ナレッジもスキルも、両方とも .md ファイルに書くよね？」

「knowledge/ と commands/、どう使い分けるの？」

✅ 一言でいうと

ナレッジ = 読む情報（知識）
スキル = 実行する手順（やり方）

前者は教科書、後者は料理レシピ。

🍕 ピザ屋で例える — いちばん分かりやすい

店員の仕事を分解すると、この2つが見える

📚 ナレッジ（読むもの）

・メニュー表
・材料辞典（粉・チーズ・ハーブの知識）
・常連さんの好みリスト
・過去のクレーム集

→ 全部「頭に入れておく情報」。それ単体では動かない。

📋 スキル（実行するもの）

・ピザを焼く手順書
・注文受付マニュアル
・クレーム対応フロー
・閉店チェックリスト

→ 全部「こう動く」の手順。呼び出されると実行される。

ポイント

店員は まず材料辞典（ナレッジ）を読んで 食材を理解し、ピザ手順書（スキル）に沿って 焼く。
ナレッジだけでは動けない、スキルだけでは品質が出ない。両方必要。

📂 VOYAGE号のフォルダ構造で視覚化

実際に見ると一目瞭然

名前の付き方のクセも覚えとくと便利

・ナレッジ = 「〇〇-db.md」「〇〇-philosophy.md」など 「調べ物」っぽい名前
・スキル = 「〇〇-park.md」「handoff」「nice-dive」など 「動作」っぽい名前

🎯 3つの判別基準

迷ったらこれで判断

質問ナレッジスキル
「/〇〇」で呼び出す？❌ しない✅ する
AIが「自分で参照」する？✅ する❌ ユーザーが起動
中身に「ステップ1, 2, 3...」がある？❌ ない（情報の羅列）✅ ある（手順）
書き方散文・リスト・データ手順書・アクション
人間で例えると辞書・教科書・資料室マニュアル・レシピ・チェックリスト

🤝 ナレッジとスキルは一緒に働く

「入れ子構造」で使われるから、最初は境界が曖昧に見える

実際の業務では、スキルの中でナレッジを参照することがほとんどです。この「入れ子構造」が混乱の元凶。

# banner-park.md（スキル・手順書）の中身例

Step 1: knowledge/pak-philosophy.md を読む ← ナレッジを参照
Step 2: ユーザーに商品情報を聞く
Step 3: clients/lutemo.md を読む ← ナレッジを参照
Step 4: コピーを3案生成
Step 5: バナー画像を生成
Step 6: ユーザーに確認
この関係性を理解すると

スキルは「手順書」なので、中で色々な「辞書（ナレッジ）」を読みに行く。
料理人が「パスタを茹でる手順書」を見ながら「食材辞典」で塩の量を確認するのと同じ。

✅ 手順 = スキル / ✅ 知識 = ナレッジ / ✅ 両方あって品質が出る

この島の残りで学ぶこと
ここからは「スキル」の設計に集中します。

SCRIPT VS SKILL

「考える必要ある？」で分ける

全てをAI（Skill）でやる必要はありません。考える必要がない定型処理は、Pythonスクリプトの方が速い・安い・確実です。

なぜ大事？

電卓で計算できることをわざわざ数学教授に頼む人はいない。「AIじゃなくてもできること」にAIを使うと、コスト・速度・信頼性の全てで損をする。判断基準は「考える必要があるか？」だけ。

処理の内容判断理由
ファイル名の一括リネーム🐍 スクリプトルールが決まっている。考える余地なし
CSVの集計・グラフ化🐍 スクリプト計算は機械の仕事。LLMに計算させると間違う
画像のリサイズ・形式変換🐍 スクリプトピクセル計算に思考は不要
広告コピーの生成📋 スキル「刺さるコピー」は判断力が必要
競合分析レポート作成📋 スキルデータの解釈・洞察に思考力が要る
ブランドガイドに沿ったチェック📋 スキル「ガイドの意図」を理解する必要がある
Amazon商品タイトルの最適化📋 スキル検索意図 × 訴求力の判断が必要
FBA在庫数の一括更新🐍 スクリプト数値の計算・転記。考える余地なし
🏭 工場の例

ネジを締める作業 → ロボットアーム（正確・速い・安い）
新製品のデザイン → 人間のデザイナー（創造力が必要）

定型作業をデザイナーにやらせるのは「もったいない」。AIでも同じ。

⚓ VOYAGE号のスクリプト例

sync.sh — gitの自動同期（pull/push）
update_crew_map.py — クルーマップ自動更新
slack_feedback_bot.py — Slackフィードバック収集

どれも「考える」必要がない。決まったルールで動くだけ。

SKILL ANATOMY

Skillの基本構造 — 5つのパーツ

Skillは .claude/commands/ フォルダに置くMarkdownファイルです。/skill-name で起動できます。

Skillの5パーツ

① いつ使うか（Trigger）
 → どんな時にこのスキルを起動するか。条件を明記

② 実行手順（Steps）
 → Step 1, 2, 3... の順番で。迷う余地をなくす

③ 出力フォーマット（Output）
 → 「こういう形式で出す」のテンプレート

④ 注意点（Constraints）
 → やってはいけないこと、気をつけること

⑤ ナレッジ参照（Knowledge）
 → 「先にこのファイルを読んでから始める」の指定
テンプレート

starter-kit に skill-template.md があります。ミッションで使います。

SHOT DESIGN

Skillの品質を爆上げする「Shot」の技術

Skillに「こういうアウトプットを出して」の具体例（Shot）を入れると、精度が劇的に上がります。
ポイントは「数より多様性」。

🎯 3つのShotで90%カバーする

10個の似た例より、3つの多様な例

1

正常系の例（典型的なケース）

「こういう入力が来たら、こういう出力を出す」の王道パターン。料理で言えば「完成写真」。

2

異常系の例（エラー・例外ケース）

「こういうダメな入力が来たら、こう対処する」のパターン。料理で言えば「焦がした写真」。

3

境界ケースの例（判断に迷うケース）

「微妙な判断が求められるとき、こう判断する」のパターン。料理で言えば「ギリギリOKの焼き色」。

👨‍🍳 料理の例

レシピに「完成写真」が1枚あるだけで、文章だけのレシピより格段にわかりやすくなる。

さらに「失敗例」と「ギリギリOK」の写真があれば、判断基準がクリアになる。

⚓ VOYAGE号の実例

banner-park のSkillには:
✅ 良いバナーの例（N1に刺さるコピー）
❌ ダメなバナーの例（機能説明だけ）
🔶 微妙な例（方向性は合ってるけど弱い）

この3パターンがあるから、AIが「何が良くて何がダメか」を判断できる。

SKILL EVOLUTION

Skillはどう育つのか — VOYAGE号の成長物語

Skillは最初から完成形ではありません。使いながら育てていくもの。
バナー制作Skill「Banner Park」の成長過程を見てみましょう。

v1

最初は「とりあえず作れる」だけ

「バナーを作って」と言えば何か出てくるレベル。でもクオリティはバラバラ。人間で言えば「初日のバイト」。

v3

「手順書」ができた

Research → Concept → Banner の3ステップが定義された。毎回同じ順序で考えるから、品質が安定し始めた。人間で言えば「業務マニュアルを覚えた社員」。

v5

「品質チェック」が入った

N1の語彙が入っているか、コンセプトと一致しているか…のチェックゲートが追加された。人間で言えば「先輩のレビューが入るようになった」。

v7

「ベクトル品質ゲート」が入った（現在）

生成されたコピーが、N1の需要ベクトルとどれくらい近いかを数値で測定。0.7以上なら合格、0.5以下なら自動で書き直し。人間で言えば「数字で判断できるベテラン」。

Skillの育て方のコツ

最初から完璧を目指さない。まず動くものを作り、使ってみて、ダメだったところを直す。この「使う → 直す」サイクルが Skill を育てる。

SUMMARY

Island 04 まとめ

🛠️
3つの持ち帰り
① 考える必要がない処理はスクリプトに、考える処理はSkillに
② Skillは5パーツ構成。Shot（具体例）を3つ入れると精度が劇的に上がる

🏁 チェックポイント — この島で覚えること

  • Script（考えない処理）とSkill（考える処理）を使い分ける
  • Skillの5パーツ: Trigger / Steps / Output / Constraints / Knowledge
  • Shotは「正常・異常・境界」の3パターンで多様性を確保する
  • Skillは最初から完璧でなくていい。育てる前提で作り始める

📖 この島の航海用語

Script（スクリプト）

考える必要がない定型処理。PythonやBashで書く。速い・安い・確実。

Skill（スキル）

AIに考えさせる手順書。.claude/commands/に置くMarkdownファイル。

commands/

Skillファイルを置くフォルダ。/skill名 で起動できる。

Shot設計

Skillに入れる具体例。正常系（成功例）・異常系（失敗例）・境界（判断に迷う例）の3パターン。

Skill成長

v1→v3→v5→v7のように、使いながら品質を上げていくプロセス。

品質ゲート

アウトプットの品質を数値で判定する仕組み。基準未達なら自動でやり直し。

🛠️ はじめてのSkillを書こう

自分の業務で「毎回やっている繰り返し作業」を1つ選び、Skillのテンプレートに当てはめるハンズオン。

ゴール

starter-kit の skill-template.md を使って、自分の業務用Skillを1つ完成させる

  1. 自分の仕事で「毎週やっている繰り返し作業」を1つ選ぶ（例: 週次レポート作成、定例MTGの議事録整理）
  2. それが「Script向き」か「Skill向き」か判断する。考える要素があるならSkill
  3. Claude Codeで以下を実行:
    starter-kit/skill-template.md を読んで、
    以下の業務をSkillに変換してください。

    # 業務内容
    - タスク名: [例: 週次レポート作成]
    - 頻度: [例: 毎週金曜]
    - 手順: [箇条書きで3-5ステップ]
    - 注意点: [やってはいけないこと]

    # Shotを3つ入れて
    - 正常系: [良いアウトプットの例]
    - 異常系: [ダメなアウトプットの例]
    - 境界: [判断に迷うケースとその判断基準]
  4. 出力されたSkillファイルを読んで、自分の業務に合っているか確認・修正する
所要時間: 約25分
