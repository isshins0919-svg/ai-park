AIエージェント設計入門
複数AIを組み合わせるアーキテクチャ

1つのAIに全部やらせるのではなく、「専門家チーム」を設計する

2026年4月3日 | 講師：澤田一進

01

エージェントとは何か

AIエージェント = 特定の専門性を持つAI担当者

1-a. 1つのAIに全部やらせる vs 専門家チームに分担

単一AIアプローチ
  • 1つのAIに「記事を書いて、チェックして、修正して」と全部お願い
  • 手軽に始められる
  • タスクが複雑になると精度が落ちる
  • 1つのミスが全体に波及する
VS
専門家チームアプローチ
  • フック専門家、信頼構造専門家、CTA専門家がそれぞれ担当
  • 各AIが自分の専門領域に集中
  • 並列実行で高速化
  • 1つのミスが他に影響しない

1-b. 人間の組織と同じ発想

会社組織には営業部・マーケ部・開発部があります。なぜ全員が「なんでも屋」ではないのか？ 専門性を持つことで品質が上がり、分担することでスピードが上がるからです。

AIエージェントも全く同じ。Anthropic（Claude開発元）の設計思想は明確です：

Anthropicの設計原則

1 agent = 1 task

1つのエージェントには1つの仕事だけ。複数の仕事を持たせた瞬間、判断の質が下がる。

🏢
人間の組織

営業部 / マーケ部 / 開発部 / 法務部 / 経理部 — 各部署が専門領域を持ち、連携して成果を出す

🤖
AIの組織

フック分析Agent / 信頼構造Agent / CTA Agent / 薬機法Agent — 各エージェントが専門評価を持ち、統合判定する

🎼
オーケストラの指揮者

バイオリン / チェロ / フルート — 各楽器が最高の演奏をし、指揮者が全体をまとめる。AIシステムも同じ構造

02

エージェント設計の4要素

Purpose / Goals / Behavior Rules / Constraints

2-a. すべてのエージェントに必要な4つの定義

エージェントを作るとき、最初に決めるのは4つだけ。これが曖昧だと、エージェントは迷子になります。

🎯
1. Purpose（存在意義）

このエージェントは何のために存在するか。「なぜこのAIが必要か」を一文で説明できなければ、そのエージェントは不要。

📋
2. Goals（目標）

具体的に何を達成するか。数値目標があると判断がブレない。「hook_score ≥ 75」のように定量化する。

📐
3. Behavior Rules（行動規範）

どのように振る舞うか。評価軸、出力フォーマット、判定基準を明文化する。

🚫
4. Constraints（制約）

やってはいけないこと。越境禁止ライン。「フック以外は評価しない」のように明確に線を引く。

2-b. 実例：記事フック君の設計

実例
記事フック君 — 記事LPの冒頭200文字を評価するエージェント
設計要素 定義
Purpose 記事LPの冒頭200文字のフック強度を評価する
Goals hook_score を算出（目標 ≥ 75）
Behavior 4軸評価：悩み解像度 × N1投影度 × 感情温度 × 数字パワー
Constraints フック以外の評価はしない。CTA・信頼構造には言及しない
なぜConstraintsが重要か

フック君がCTAについてコメントし始めると、CTA専門のエージェントと判断が矛盾する。
「越境禁止」は品質を守る最も重要なルール。

04

品質を上げる設計原則

実運用から得た5つのルール

4-a. 5つの設計原則

原則 1：1エージェント = 1次元

フック君はフックだけ、信頼君は信頼だけ。越境した瞬間、判断が濁る。
人間の組織でも同じ — 営業マンが経理業務をやり始めたら、どちらも中途半端になる。

原則 2：スコアリングは重み付き

すべての評価軸が同じ重要度ではない。コンバージョンへの影響度で重み付けする。

実際の重み付け例

mCVR_score = hook × 0.40 + narrative × 0.35 + cta × 0.25

フックが最重要（0.40）なのは、読まれなければ他の要素が意味をなさないから

原則 3：GO / REVISE / BLOCK の3段階判定

曖昧な「まあまあ」を排除する。判定は3つだけ。

GO — 配信OK REVISE — 修正が必要 BLOCK — 配信不可
原則 4：制約は後入れ

最初から制約だらけにすると、AIが萎縮して平凡なアウトプットしか出さない。
まず自由に作らせて、後から薬機法・景表法等の制約を入れる。
創造性と安全性を両立するための順序設計。

原則 5：Hard Block ルール

法規違反は点数に関係なく即停止。フックがどんなに良くても、薬機法違反なら BLOCK。

Hard Block の例
  • 薬機法違反表現（「治る」「効果がある」等の断定）→ 即BLOCK
  • 景品表示法違反（根拠のない「No.1」表記）→ 即BLOCK
  • 個人情報の露出 → 即BLOCK

4-b. 原則の全体像

設計原則が組み合わさった判定フロー
STEP 1 — 各エージェントが独立評価（原則1）
hook_score: 82 | narrative_score: 71 | cta_score: 68
STEP 2 — 重み付きスコア算出（原則2）
mCVR_score = 82×0.40 + 71×0.35 + 68×0.25 = 74.65
STEP 3 — Hard Blockチェック（原則5）
compliance_violations: 0 → Hard Block なし
STEP 4 — 3段階判定（原則3）
mCVR_score 74.65 < 75 → REVISE（CTA改善を推奨）

07

設計の始め方

今日から始められる5つのステップ

7-a. ゼロからのロードマップ

STEP 1
まず手動で10回やる

自動化したいタスクを、まず自分で10回やる。するとパターンが見えてくる。「毎回同じことを考えている」「この判断は定型化できる」というポイントがエージェント化の候補。

STEP 2
パターンをプロンプトに書き出す

発見したパターンを「あなたは〇〇の専門家です。〇〇を評価してください」というプロンプトに言語化する。これが最初のエージェントの原型。

STEP 3
Projects機能で「専門家」として育てる

Claude の Projects に入れて、ナレッジを追加し、何度もフィードバックを繰り返す。「ここは違う」「これは良い」を繰り返すことでエージェントが育つ。

STEP 4
2つ目のエージェントを追加する

1つ目が安定したら、別の軸を担当する2つ目を追加。ここで初めて「役割分担」が生まれる。1つ目と2つ目が矛盾しないよう、Constraintsを明確にする。

STEP 5
徐々にチームを拡大

3体、4体と増やしていく。司令塔（指揮者）を置くかどうかは、5体を超えたあたりで検討する。焦らず、1体ずつ確実に育てることが最重要。

最も大事なこと

いきなり8体のエージェントシステムを作ろうとしないこと。
澤田のシステムも、最初は「1つのプロンプト」から始まった。1体ずつ育てて、必要に応じて増やす。
この積み重ねが、気づけば20体のAIチームになる。

7-b. 始めるための3つの問い

❓
問い 1：毎週、繰り返しやっている作業は？

「レポート作成」「競合チェック」「記事の品質確認」— 繰り返しているものこそエージェント化の最有力候補。

❓
問い 2：その作業の「判断基準」を言語化できるか？

「良いフックとは何か」を言語化できれば、それがエージェントのBehavior Rulesになる。言語化できないものはまだ手動でやる。

❓
問い 3：間違えたとき、すぐ気づけるか？

AIの出力が間違っていたとき、人間がすぐ気づけるタスクから始める。気づけないタスク（法務判断等）は後回し。

MESSAGE

1つのAIに全部やらせる時代は終わった。
専門家チームを設計する。それがAI活用の次のステージ。
まずは1体、育ててみてください。
