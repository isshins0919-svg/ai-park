なぜ作ったか
記事LPのフィードバックは 属人化・時間コスト・品質ムラ が課題だった。 マーケターが「フィードバックください」とURLを貼るだけで、競合データ付きの本格的なLPレビューが即時返ってくる仕組みが必要だった。
誰が使うか
ヨミテ社のマーケターが Slack の #yomite_ai-kiji_fb チャンネル に記事LP のURLを投稿すると自動でフィードバックが返ってくる。 人手不要・24時間対応。
何が返ってくるか
HTML レポート（記事スクショ＋FB の左右2カラム）、または Slack Canvas。 フック評価・構成分析・画像診断・競合比較・今週伸びてる記事3本URL提案 まで含む。
競合インサイトの源泉
DPro MCP（広告費増加ランキングデータ）から4ジャンルのTOP記事LPを取得・Playwright解析し、 dpro_benchmark.json として蓄積。FB生成時に自動参照。

Development Journey
開発の歩み
v1 — 起点
Slack ポーリング × テキスト抽出の原型
Slack の特定チャンネルを2分おきにポーリングし、URLを含む「フィードバック」投稿を検知してPlaywrightでテキスト抽出。 Claude Sonnet にLP本文を渡してテキストベースのFBを返す最初の原型。
Slack SDK Playwright Claude API
v2〜v3
画像Vision対応 × DOM順セクション解析
テキストのみから Claude Vision 対応へ。DOM順に画像をテキストとセットで渡すことで 「ビジュアル×コピーのセット評価」が可能に。18パートの構造分析プロンプトに昇華。
Claude Vision DOM順解析 18パート分析
v4〜v5
HTMLレポート生成 × Slack Canvas 対応
フィードバックをSlackテキストとして返すだけでなく、 記事LPのスクリーンショットとFBを左右に並べたHTMLレポートを自動生成してファイル添付。 アップロード失敗時の Canvas → テキストフォールバック設計。
HTML報告書 Slack Canvas フォールバック設計

OpenCV+ThreadPoolExecutor で動画サムネイルを8並列取得。 DPro MCP から4ジャンルTOP記事LPを取得・Playwright分析してbenchmark.jsonに蓄積。 FB末尾に「今週伸びてる競合記事3本URL＋盗むべきポイント」提案セクションを追加。

Technical Challenges
解決した技術的課題
01
UnicodeEncodeError — APIキーに日本語が混入する謎のバグ
zsh -i（インタラクティブシェル）で環境変数を取得していたため、 ターミナルの「Restored session: 2026年 3月31日…」という復元メッセージがAPIキーのstdoutに混入。 zsh -l（ログインシェル）＋source ~/.zshrcの明示実行に変更して根本解決。 副次的にhttpxのASCIIヘッダーパッチも追加。
Root Cause Found
02
ターミナルを閉じるとプロセスがSIGHUPで死ぬ
nohup zsh -l -c "..." でSIGHUPを無視してバックグラウンド実行。 PIDを/tmp/fb_bot.pidに書いて再起動・停止を管理可能に。
Ops Fixed
03
DPro MCPはClaudeセッション内でしか使えない（Pythonから直接呼べない）
MCPツールはClaudeセッション内でのみ使用可能。 「セッション内でDPro → LP解析 → benchmark.json更新」を手動で実施しておき、 Pythonボットはファイルを読み込むだけの設計に分離。
Architecture Decision
04
動画サムネイル取得がシリアル処理でボトルネック
ThreadPoolExecutor(max_workers=8) で全動画のサムネイル取得を並列化。 失敗した動画はスキップして続行する設計で安定性を担保。
