FEEDBACK LOOPS

フィードバックループ = 「やる → 測る → 直す」の自動化

前回の勉強会で森田さんが「フィードバックしていってサイクルを作っていくことが一番大事」と言っていました。これがフィードバックループです。

なぜ大事？

テスト勉強で「問題を解く → 丸つけしない → また同じ範囲を解く」を繰り返す人はいない。丸つけ（評価）して間違いを直す（改善）から成績が上がる。AIも同じ。実行→評価→改善のサイクルがないと、何回やっても同じ品質のまま。

🏋️ 筋トレの例

❌ 毎日なんとなく腕立て → 3ヶ月後も変化なし

✅ 記録つけて → 重量を少しずつ上げて → フォーム動画で確認 → 着実に成長

「記録 → 分析 → 調整」のサイクルが成長を生む。

📊 営業の例

❌ テレアポ100件 → 結果を振り返らない → 同じトークで次の100件

✅ テレアポ100件 → 「どのトークで何件取れたか」分析 → トーク改善 → 次の100件

PDCAが回る営業チームは自然と数字が上がる。

THREE LOOPS

VOYAGE号が目指す3つのフィードバックループ

🔧 ループA: スキル自動改善

「このSkillのどのステップが弱いか」を自動で見つけて直す

Skillを実行 → 結果の品質スコアを記録 → 3回分の傾向を分析
→ 「Step 3で品質が落ちる傾向」を発見 → Step 3だけ自動修正
🍰 パティシエの例

新人パティシエが毎日ケーキを焼く。
師匠が「今日のスポンジはちょっと硬い」「今日は完璧」と毎回フィードバック。
3回分のフィードバックを見ると「卵の泡立てが甘い日に硬くなる」パターンが見える。
→ 泡立て工程だけを重点的に改善。

⚓ VOYAGE号での実装予定

各Skillの実行結果を logs/skill_metrics.json に記録。
/park-kaizen 実行時にメトリクスを分析 → 弱いステップを特定 → 自動で改善提案。

Phase 3で実装予定

📈 ループB: CRパフォーマンスフィードバック

「実際に売れたCR」のデータが、次のCR生成の精度を上げる

CRを生成 → 広告配信 → DPro MCPで実績取得
→ ベクトルDBに「勝ち/負け」タグ付与 → 次回生成時に「勝ちゾーン」を参照
🎯 ダーツの例

1投目: 的の左上に当たった → 「右下にずらす」
2投目: 的の中心付近 → 「この力加減を記憶する」
3投目: ブルズアイ

毎回の結果を次に反映するから精度が上がる。結果を見ないで投げ続けても、いつまでも当たらない。

⚓ VOYAGE号での実装予定

① DPro APIでCR実績を取得（cost_difference, digg_rate等）
② 実績 × CRのベクトルを紐付けて蓄積
③ 「勝ちCR」のベクトル重心を計算
④ 新CRは「勝ちゾーン」に近い方向で生成

Phase 4で実装予定

🚦 ループC: コンテキスト品質監視

「AIがパンクしそう」を自動検知して、手遅れになる前に対処

セッション中のNG回数をカウント → 3回以上NGが続いたら警告
→ サブエージェントに委譲 or セッション分割を提案
🚗 車のダッシュボードの例

ガソリンが少なくなったら警告ランプが光る。エンジン温度が上がりすぎたら警告が出る。

壊れる前に教えてくれるから、事故を防げる。
AIも同じ。パンクする前に「そろそろ限界です」と教えてほしい。

→ 森田さん自身が「あ、やっちゃいけないことしてた」と気づいた。

これを自動で検知できるようにするのがループC。

REFLECTION

リフレクション — 「振り返って」は曖昧すぎる

リフレクションは「AIに自分の出力を振り返らせて修正させる」手法。ただし「品質を改善して」のような曖昧な指示では機能しない。

❌ NG — 曖昧なリフレクション

「品質を振り返って改善して」
「もっと良くして」
「前のと比較して」

→ AIは「何をどう改善すればいいか」がわからないから、表面的な変更しかしない

✅ OK — 具体的なリフレクション

「各ステップの出力を比較して、以下の異常を検出して」

・トーンの急変（前ステップと感情スコアが ±30% 以上）
・情報の欠落（前ステップにあった必須項目が消えている）
・矛盾（前ステップの結論と逆の主張が出現）

「異常があったステップのみ再実行」

📝 作文添削の例

❌ 先生「もっと良くしましょう」→ 何を直せばいいかわからない

✅ 先生「①主語と述語がねじれている箇所 ②同じ接続詞の3回以上使用 ③段落冒頭の繰り返し表現をチェックして修正して」

チェック項目が具体的だから、改善が的確になる。

🏃 スポーツコーチの例

❌ コーチ「もっと上手くなれ」

✅ コーチ「腕の振りが小さい。肘の角度を90度に。腕を耳の横まで上げろ」

測定可能な改善ポイントを示すのがリフレクションの本質。

前回の勉強会との接続

森田さんが「アウトプットに対して『これ違うわ』って具体でフィードバックしちゃってた。本質的に何がいけてないかが模索されずにアウトプットだけ変える努力をしてた」と自己反省していた場面がまさにこれ。

表面（アウトプット）を直すのではなく、根本（なぜそうなったか）を直す。それがリフレクションの正しい使い方。

MULTI-LAYER DEFENSE

多層防御 — 「AIが間違えても大事故にならない」設計

AIは間違えます。これは前提。だから「間違えても被害が出ない仕組み」を複数の層で作る。

多層防御の3層

第1層: スクリプト検証（機械的チェック）
 → 入力値フォーマット確認、禁止ワードチェック等
 → LLMを使わない = 100%確実な防波堤

第2層: AI自己チェック（リフレクション）
 → 自分の出力に矛盾・異常がないか確認
 → 精度は高いが100%ではない

第3層: 人間の確認（最終ゲート）
 → 金額が大きい操作、公開する内容、不可逆な操作
 → 「送信ボタン」を押すのは人間

原則: 検証はスクリプトで。判断はAIで。最終承認は人間で。
🏦 銀行ATMの例

第1層: カード + 暗証番号（機械的認証）
第2層: 異常取引検知AI（パターン照合）
第3層: 高額送金は窓口で本人確認（人間チェック）

3層全部突破されないと被害が出ない。

⚓ VOYAGE号の多層防御

第1層: rules/security.md でAPIキー直書き・強制push等を禁止（機械的ルール）
第2層: gate-quality-crew が品質ゲートでPASS/FAILを判定（AIチェック）
第3層: Slack投稿・メール送信は人間が最終確認（人間ゲート）

前回の勉強会でも「アップサイダーの上限10万円で止まった」という話があったが、あれも多層防御の一つ。
