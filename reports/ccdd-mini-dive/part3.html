コンテキストウィンドウ
— AIの脳の限界

何を入れて、何を入れないか。この設計が全品質を決める

所要時間: 約10分

📌
Part 2の振り返り
指示は5要素で書く。考える必要がない処理はスクリプトに。Skillは使いながら育てる。
今回は「なぜAIが嘘をつくのか」「なぜ指示を忘れるのか」の根本原因に迫る。
WHAT IS CW

コンテキストウィンドウ（CW）= AIの「今の頭の中」

CWは「AIが一度に処理できる情報量の上限」。人間の短期記憶（ワーキングメモリ）と同じです。
前回の勉強会で森田さんが「コンテキストウィンドウがいっぱいになっちゃう」と言っていたのがこれ。

📚 机の上の例

仕事の机を想像してください。

机の上に書類を3枚広げて作業する → 集中できる、全部見える
机の上に書類を50枚広げて作業する → 探すのに時間がかかる、大事な書類が埋もれる、間違った書類を見て判断する

CW = 机の広さ。広げすぎると作業効率が落ちる。

🎒 通学カバンの例

全教科の教科書を毎日持っていく → 重い、使わない教科書ばかり
時間割を見て必要な教科書だけ持っていく → 軽い、すぐ出せる

「必要なものだけ持つ」がCW管理の本質。

WHEN CW OVERFLOWS

CWが溢れると起きる4つの症状

前回の勉強会で森田さんが「ハルシネーション」と言っていたのは、CWが圧迫されたときの症状の一つです。

🚨 CW圧迫の4症状

これが出たら「頭がパンク」しているサイン

症状何が起きているか人間で例えると
① ハルシネーション
（嘘をつく）		答えを知らないのに、もっともらしい嘘を生成する徹夜明けで記憶が曖昧なのに「たしかそうでした」と答えてしまう
② 指示忘れ最初に伝えた指示が、途中から無視される10個の頼み事をされて、後半の3個を忘れる
③ ループ悪化同じミスを何度も繰り返す。修正しても元に戻る疲れてるときにタイプミスを直しても、また同じ場所で間違える
④ 精度低下全体的にアウトプットの質が落ちるマルチタスクのしすぎで全部の仕事が雑になる
なぜこうなるのか

CWは「今、頭の中に入っている全情報」。ここに指示、会話履歴、読んだファイル、生成中のテキスト…が全部入る。容量が埋まるほど「古い情報」や「優先度の低い情報」が実質的に見えなくなる。消えるわけではないが、AIの注意が行き届かなくなる。

TIER DESIGN

CWの4階層管理 — VOYAGE号の設計

CWを効率よく使うために、VOYAGE号では情報を4つの階層（Tier）に分類しています。

CW管理の4階層

Tier 0: CLAUDE.md + rules/
 → 全メッセージに必ず入る。1行でも減らしたい
 → 例: セキュリティルール、個人情報保護ルール

Tier 1: Read（knowledge/ 等）
 → 読んだらセッション中ずっと残る。必要な時だけ読む
 → 例: マーケ哲学、フックDB、クライアント情報

Tier 2: commands/（Skills）
 → /skill で起動すると親に読み込まれる。重いスキルは注意
 → 例: /banner-park, /kiji-cko

Tier 3: agents/（サブエージェント）
 → 独立空間で動く。親のCWを消費しない
 → 例: kiji-hook-crew, movie-arc-crew

💡 重い処理はTier 3（エージェント）に任せる = 親のCWを守る
🏢 オフィスの例

Tier 0 = 会社の全社ルール（全員が常に従う）
Tier 1 = 部署の資料棚（必要な時だけ取り出す）
Tier 2 = 会議に持ち込む資料（持ち込みすぎると机が溢れる）
Tier 3 = 別室で作業する専門チーム（本部の会議室を使わない）

🎒 遠足の例

Tier 0 = お弁当・水筒（絶対持っていく）
Tier 1 = 地図・しおり（目的地で見る）
Tier 2 = おやつ（持っていくけど重い）
Tier 3 = 大型荷物はバスのトランクに（自分のリュックに入れない）

AGENT SPLITTING

エージェント分割 — いつ分けるか

前回の勉強会で森田さんが「1体に全部やらせると制御できなくなる」と言っていたのは、CWの限界が原因です。
解決策は「分ける」こと。専門タスクごとにエージェントを分割すれば、それぞれのCWが守られます。

✂️ 分割の判断基準 — 3つのNG信号

これが出たら、1つのセッションに詰め込みすぎ

!

ループ3回以上NG

同じミスを修正しても3回繰り返す → タスクが複雑すぎるか、CWが圧迫されている

!

指示忘れが発生

最初に伝えた条件が途中から無視される → CWの初期部分の情報に注意が行き届いていない

!

ハルシネーションが発生

存在しない情報を「ある」と言い始める → 脳がパンクしている明確なサイン

🎬 映画制作の例

監督1人が脚本・撮影・照明・音声・編集を全部やったら？
→ 撮影中に脚本の直しが入って、照明の指示を忘れる

各部門にプロを置く = エージェント分割
監督（= オーケストレーター）は全体の指揮だけに集中。

⚓ VOYAGE号の分割例

記事LP制作の場合:
/kiji-cko（オーケストレーター）が指揮
→ kiji-hook-crew（フック評価）
→ kiji-trust-crew（信頼構造評価）
→ kiji-arc-crew（感情アーク評価）
→ kiji-offer-crew（オファー評価）

各クルーは独立CW（Tier 3）で動くから、親のCWを使わない。結果だけ返す。

設計パターン役割CWへの影響
commands/
オーケストレーター		ユーザーと対話。全体の指揮を取る親のCWを使う（Tier 2）
agents/
サブエージェント		専門タスクを独立で実行独立CW（Tier 3）。親に影響なし
前回の勉強会との接続

森田さんの「CEO → CMO → サブエージェント」構造がまさにこれ。CEOは全体の方向性だけ考え、CMOはクリエイティブ戦略だけ考え、各サブエージェントは自分の専門だけに集中する。それぞれのCWに余裕があるから、品質が高い。

PLAN MODE

なぜ「プランモードで考え、次のセッションで実行する」のか

前回の勉強会で森田さんが強調していた「プランを立てたセッションで実行はしない」。
これもCWの話です。

CWの消費イメージ

【プランニング】
 要件整理、設計、シミュレーション… → CWの60%を消費

【実行】
 コード書き、ファイル作成、テスト… → CWの50%が必要

【両方同じセッション】
 60% + 50% = 110% → CWオーバーフロー → ハルシネーション発生

【分離する】
 セッション1: プラン（60%使用）→ ハンドオフ
 セッション2: 実行（50%使用）→ フレッシュなCWで高品質
🏗️ 建築の例

設計士が設計図を描きながら同時にレンガを積む人はいない。

設計は設計、施工は施工。分けるから両方の品質が高い。
AIのプランモードと実行モードも同じ理屈。

📝 試験勉強の例

勉強計画を立てる日と、実際に勉強する日を分けた方が効率がいい。

計画を立てながら勉強すると、どちらも中途半端になる。
ハンドオフ = 勉強計画をノートに書いて、翌日の自分に渡す。

SUMMARY

Part 3 まとめ

🧠
3つの持ち帰り
① CWは有限。詰め込みすぎるとAIは壊れる（嘘、忘却、精度低下）
② 情報をTier 0〜3に分類して、必要なものだけCWに入れる
③ 重い処理はサブエージェント（Tier 3）に任せ、プランと実行はセッションを分ける
