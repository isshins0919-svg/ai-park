Chapter 03

エンベディング — AIに「意味」を理解させる技術

「エンベディングを知らないとクリエイティブは何もできないぐらいの技術なのに、みんな知らない。これはアドバンテージ。」
— 森田新之助（パク）

1 エンベディングとは

テキストデータを3,072次元のベクトル空間に変換する処理。「おはよう」なら「朝っぽい」「挨拶っぽい」「お別れっぽくはない」——こういった意味の成分を数値化する。

  • 処理なし: AIは言葉を単なるパターンとして扱う。「こう言われたらこう返す」レベル
  • 処理あり: AIが意味を深く理解する。「身長」と「伸び伸び」が似た概念だと分かる

2 エンベディング処理の効果（Before / After）

SNSタイプ診断バナーの例:

  • 処理なし: ゲームっぽい仕上がり。施策の意味が分かっていない。人間がキャッチコピーを考え、ディレクションしないとダメ
  • 処理あり: AIが意味を理解し、どう出せばいいか分かる。クリエイティブの質が圧倒的に向上

3 エンベディング2（2026年3月登場）

テキストだけでなく、イメージ・ビデオ・オーディオもひとつのベクトル空間に統合できる技術。マルチモーダル・エンベディング。

  • 提供元: OpenAI / Google（Gemini）のAPIで利用可能
  • コスト: 100万行 × 500次元程度で数万円前半。比較的安価

実務での使い方

200本の売れてる動画と100本の売れてない動画を全部ベクトル化 → 売れてる広告の共通要素を抽出。人間が200本全部見て分析したのと同じ結果がAIから出てくる。ベクトル空間内で「需要ベクトルが高い」「オンリーワンベクトルが高い」と指定できるため、独自性の高いコンセプトをAI自身が生み出せるようになる。

Chapter 04

データの質 > データの量

量の目安

200本あれば十分。
売れてるクリエイティブ200本を丁寧にベクトル化すれば、十分な精度でスコアリングが可能。

質がキモ

スコアリングが大事。
「どれが売れてる」「どれが売れてない」まで全部ラベリングすると、売れてる方に寄せて作ってくれる。ラベルなしだと「全部足して割った平均」になる。

データベース構築 = 最重要インフラ

人間と同じで、AIも「何も知らない状態」から始めると良いアウトプットが出ない。ビタミンCクレンジングのマーケティングを頼まれたら、まずリサーチする。AIも同じ。

  • 分析基盤を作る子と分析をする子を分ける
  • 売れてるクリエイティブを洗い出し → エンベディング → AIが「売れるパターン」を意味として理解
Chapter 05

エージェント・オーケストレーション

まず1体を育てる

最初からオーケストレーションを構築するのではなく、まずは狙い通りに動く1体を作る。人間がオーケストレーターとして複数エージェントを束ねる。5体ぐらいになったらマネージャーエージェントを作る。

1体を精密に育てる → 人間がオーケストレーション → 5体になったらマネージャーを作る → ホールディングス型へ

新エージェントは上司にレビューさせる

新しく作ったエージェントは、CMOなどの上位エージェントにレビューしてもらう。質の高いレビューが得られ、エージェント作成コストが下がる。最初の優秀なやつだけ何回か作れば、あとは楽になる。

粒度の決め方

コンテキストウィンドウ（100万トークン）を基準に考える。

  • タスクが少なすぎ → 1体に統合してOK
  • 言うこと聞かなくなった → コンテキスト溢れ。分割
  • 例: 動画編集は「テロップ入れ」「カット」など5体ぐらいに分ける。1体に全部やらせない

プランモード — 考えることと作ることを分ける

「エージェントを生み出す時はプランモード。動かす時はプランモードを使わない。」
— 森田新之助（パク）

なぜプランモード必須なのか

  • ハルシネーション防止: あれこれ言いながら作るとコンテキストが溢れ、嘘をつく・いい加減な仕事をする
  • 建築と同じ: まずしっかり考えてから、めちゃくちゃしっかり作る
  • プランを立てたセッションで実行はしない: 思考力を使い切っているため。次のセッションに引き継ぐ

魔法の言葉

エージェントを作る時に言う一言:
「要件定義書、基本設計、技術要件、詳細仕様書、テスト書を作って」
これで全部作ってくれる。この「型」を知っていることが大事。

Chapter 08

フィードバックの哲学

最も重要な気づき

アウトプットに対して「いやこれ違うわ」と表面の具体的なフィードバックをしてはいけない。本質的に何がいけないのかが模索されず、アウトプットを変えるだけの努力になってしまう。

正しいフィードバックの方法

  • NG 「この色を変えて」「この文言を直して」 → 表面が変わるだけで本質が解決されない
  • OK 「なぜこうなったか」を問う → 根本原因から改善される

オーケストレーションを理解した上で指示すると、PDCAが回せるようになる。

PDCAの回し方

ビジュアルの仮説（「チェスが売れたからチェス風をもっと」）ではなく、N1とコンセプトの組み合わせだけを試す。ファンダメンタルで考える。

Chapter 09

制約は後入れ — クリエイティブの自由を守る

後入れ原則

最初から薬機法などの制約を入れすぎると、良いクリエイティブができない。

まず自由に作らせる → 後から制約チェック（薬機法等） → 制約に引っかかる部分だけ修正

実例: アポロ君（営業エージェント）も、最初は自由に作って後から制御したら良い結果に。制約を先に入れたバージョンは「だいぶ微妙」だった。
