🧠 LLM — 脳みそ

考える力そのもの。でも「脳だけ」では何もできない

LLM（Large Language Model）はAIエージェントの「脳」。テキストを理解し、推論し、生成する能力の源泉。でも脳だけでは現実世界に手出しできない。ファイルを読み書きしたり、検索したり、メールを送ったりする「手足」が必要。

なぜ大事？

どんなに優秀な人でも、手足を縛られたら仕事はできない。逆にどんなにツールが揃っていても、判断力がなければ使いこなせない。脳の能力に合った道具を選ぶのが設計の第一歩。

🏢 仕事の例

新入社員に「自由にやっていいよ」と言っても、PCのログインIDすら渡されていなければ何もできない。
逆に全ツールへのアクセス権があっても、業務知識がなければ暴走する。

🍳 料理の例

世界一のシェフでも、包丁もコンロもない部屋では料理できない。
逆に最高の調理器具を持っていても、レシピを知らなければ使えない。

🔧 Tools（MCP）— 手足

AIが「実際にやる」ための道具。接続先が増えるほど能力が広がる

MCP（Model Context Protocol）は、AIが外部サービスを操作するための統一規格。Slack送信、Google Calendar確認、ファイル操作、ブラウザ操作…AIに「手足」を与える仕組み。

なぜ大事？

ChatGPTに「メール送っといて」と言っても送れない。Claude Codeに「Slackに投稿して」と言ったら本当に投稿できる。この差がツール（MCP）。 AIが「考えるだけ」から「実行できる」に進化するかどうかの分水嶺。

📱 スマホの例

アプリがインストールされていないスマホは、電話しかできない。
LINEを入れれば連絡できる。カメラアプリを入れれば写真が撮れる。
MCPは「AIにアプリを入れる」ようなもの。

🔌 USBの例

USBは「どのメーカーのデバイスでも統一規格で繋がる」仕組み。
MCPも同じ。Slack・Google・Canva…全部同じ仕組みでAIに繋がる。
バラバラのAPIを覚える必要がない。

うちで繋いでるMCP何ができるか
Slackメッセージ送信・チャンネル読み取り
Google Calendar予定確認・作成・調整
Google Driveドキュメント検索・読み取り
Canvaデザイン生成・編集・エクスポート
DPro（動画ProMCP）広告データ検索・競合分析
Chromeブラウザ操作・スクリーンショット

⚡ Trigger — きっかけ

「いつ動くか」を決める。手動 → 自動化の進化ポイント

Triggerは「AIがいつ動き出すか」のきっかけ。人間がチャットに打ち込む「手動」から、時間指定・イベント検知の「自動」まで段階がある。

なぜ大事？

毎朝「今日の予定教えて」と打つのは面倒。「毎朝8時に自動で予定を教えてくれる」なら便利。人間が起動しなくても動くのが、ツール→エージェントへの進化。Triggerはその鍵。

⏰ 目覚まし時計の例

毎朝自分で時計を見て起きる = 手動
アラームが鳴って起きる = 自動トリガー
スマートアラームが睡眠の浅い時に起こす = 条件付きトリガー

📧 メール通知の例

自分でメールアプリを開いて確認 = 手動
新着メールが来たら通知 = イベントトリガー
重要なメールだけ通知 = フィルター付きトリガー

トリガーの種類例自動化レベル
手動（ユーザー起動）チャットで「/morning」と打つ⭐
スケジュール（cron）毎朝8時にニュースを自動投稿⭐⭐
イベント駆動Slackにメンションが来たら返答⭐⭐⭐
条件付き自動広告CPAが閾値超えたらアラート⭐⭐⭐⭐

🧭 Instructions — マニュアル

「言わなかったら間違う方向に進むか？」で書くかどうかを判断する

Instructionsは、AIの行動を方向づける「マニュアル」。CLAUDE.mdやidentity.md、rules/ に書かれている指示がこれに当たる。

なぜ大事？

「新人に仕事を教えるマニュアル」と同じ。マニュアルがなければ自己流で動いてしまう。でもマニュアルが長すぎると誰も読まない。「書かなかったら事故るか？」がYESの項目だけ書く。

指示の5要素チェックリスト
要素意味書かないとどうなるか
役割「あなたは○○です」AIが自分の立場を判断できずトーンがブレる
目的「○○を達成する」手段が目的化する（コードを書くことが目的に）
背景「この仕事の文脈は…」的外れな提案をする
制約「○○はしないで」やらかす（機密漏洩、データ削除等）
参考情報「このファイルを参照して」自分の知識だけで判断→古い情報で回答
🏗️ 建築現場の例

設計図なしで「いい感じの家を建てて」と言ったら？
→ 窓の位置も部屋の数もバラバラな家ができる。
設計図 = Instructions。「良い指示」は誤解の余地を消す。

🎯 上司への報告の例

上司に「この資料作って」と言われて「目的は？誰向け？」を聞かないと、
的外れな資料ができあがる。
AIも同じ。「背景」と「目的」がないと間違った方向に走る。

❌ NG — 曖昧な指示

「いい感じのバナーを作って」
「品質を改善して」
「ユーザーが喜ぶようにして」

✅ OK — 明確な指示

「30代女性向け・健康食品・初回半額を訴求するバナー」
「CTRが3%未満のコピーを、N1の語彙に書き換えて」
「カート完了率が70%を下回ったらアラートを出して」

📋 Skills — 業務手順書

「毎回同じ説明をするなら、手順書にしよう」がSkillの発想

Skillは「再利用可能な業務手順」。毎回チャットで長い指示を書く代わりに、一度まとめておけば `/skill名` で呼び出せる。

なぜ大事？

料理のレシピと同じ。毎回「卵を割って…フライパンに油を…」と考えるのは非効率。レシピにまとめれば、誰でも同じ品質で再現できる。AIのSkillも同じ。一度作れば何度でも使える。

🍔 ファストフードの例

マクドナルドが世界中で同じ味を出せるのは、全工程がマニュアル化されているから。
パティの焼き時間、ソースの量、包み方…全部「Skill」。
属人化していないから、誰でも同品質で出せる。

📝 チェックリストの例

パイロットが離陸前にチェックリストを確認する理由：
「毎回全部覚えているはず」は危険。手順を外出しすることで、ミスを構造的に防ぐ。
Skillも同じ。品質のバラつきを抑える仕組み。

Skillの品質を上げるコツ — Shot（具体例）の設計

Skillに「こういうアウトプットを出して」の具体例（Shot）を入れると、精度が劇的に上がる。

ポイントは「数より多様性」:

  • 正常系の例 × 1（典型的なケース）
  • 異常系の例 × 1（エラー時・例外ケース）
  • 境界ケースの例 × 1（判断に迷うケース）

3つの例で90%のケースをカバーできる。10個の似た例より、3つの多様な例の方がはるかに効果的。


料理で例えると: レシピに「完成写真」が1枚あるだけで、文章だけのレシピより格段にわかりやすくなる。さらに「失敗例」（焦がした写真）と「ギリギリOK」（焼き色薄めでもOK）の写真があれば、判断基準がクリアになる。

💾 Memory — 長期記憶

「前も同じ説明したよね？」をなくす仕組み

Memoryは「セッションをまたいで覚えておく情報」。ユーザーの好み、過去の判断基準、プロジェクトの背景情報など。

なぜ大事？

行きつけのカフェで毎回「私はカフェラテのミルク多めで…」と説明するのは面倒。常連になれば「いつもの」で通じる。AIのMemoryも「いつもの」を可能にする仕組み。なければ毎回初対面。

🏥 カルテの例

病院に行くたびに「アレルギーありますか？」「既往歴は？」と聞かれたら？
カルテがあるから、前回の情報を引き継げる。
Memoryは「AIのカルテ」。

📓 引き継ぎノートの例

バイトのシフト交代で「引き継ぎノート」がなかったら？
「○○さんが来たら冷蔵庫のケーキを取り置いてある」みたいな情報が消える。
Memory = シフト交代でも情報が消えない仕組み。

🪟 Context Window — ワーキングメモリ

AIの「今、頭の中に入っている量」。これが全ての品質を決める

Context Windowは「AIが一度に処理できる情報量の上限」。人間の短期記憶（ワーキングメモリ）に相当する。情報を詰め込みすぎると、古い情報を忘れたり、精度が落ちたりする。

なぜ大事？

会議で10個の議題を一気に話されたら、最後の方は覚えていないし、最初の議題の結論も曖昧になる。AIも同じ。情報を詰め込みすぎると「ハルシネーション（嘘をつく）」「指示忘れ」「精度低下」が起きる。だから何を入れて何を入れないかの設計が命。

🧳 スーツケースの例

30Lのスーツケースに全部詰め込もうとすると…
→ 大事なものが奥に埋もれて取り出せなくなる
→ 入りきらない荷物は落ちていく
Context Window = スーツケースのサイズ。詰め方の設計が重要。

🎒 通学カバンの例

全教科の教科書を毎日持っていく → 重い、使わない教科書ばかり
時間割を見て必要な教科書だけ持っていく → 軽い、すぐ出せる
「必要なものだけ持つ」がCW管理の本質。

CW管理の4階層（うちの設計）

Tier 0: CLAUDE.md + rules/ → 常に入っている。1行でも減らしたい
Tier 1: Read（knowledge等） → 読んだらセッション中残る
Tier 2: commands/ → /skill で読み込み。重いスキルは注意
Tier 3: agents/ → 独立空間で動く。親のCWを消費しない

💡 重い処理はTier 3（エージェント）に任せる = 親のCWを守る

設計の原則

7つの構成要素を「どう組み合わせるか」の原則。ここからが腕の見せどころ。

⚖️ スクリプト vs スキル — 考える必要ある？

「考える必要がない処理」にAIを使うのは無駄遣い

全てをAI（Skill）でやる必要はない。考える必要がない定型処理は、Pythonスクリプトの方が速い・安い・確実。

なぜ大事？

電卓で計算できることをわざわざ数学者に頼む人はいない。「AIじゃなくてもできること」にAIを使うと、コスト・速度・信頼性の全てで損をする。判断基準は「考える必要があるか？」だけ。

処理の内容判断理由
ファイル名の一括リネーム🐍 スクリプトルールが決まっている。考える余地なし
CSVの集計・グラフ化🐍 スクリプト計算は機械の仕事。LLMに計算させると間違う
広告コピーの生成🤖 スキル「刺さるコピー」は判断力が必要
競合分析レポート作成🤖 スキルデータの解釈・洞察に思考力が要る
画像のリサイズ🐍 スクリプトピクセル計算に思考は不要
ブランドガイドに沿った色チェック🤖 スキル「ガイドの意図」を理解する必要がある
🏭 工場の例

ネジを締める作業 → ロボットアーム（正確・速い・安い）
新製品のデザイン → 人間のデザイナー（創造力が必要）
定型作業を人間にやらせるのは「もったいない」。AIでも同じ。

🧮 経理の例

請求書の合計金額計算 → Excel（= スクリプト）
予算配分の判断 → 経理部長（= スキル/AI）
計算は道具に、判断は頭脳に。

✂️ エージェント分割 — いつ分けるか

1人に全部やらせたら、パンクする。人間もAIも。

1つのセッション（= 1人のAI）に複雑なタスクを全部任せると、Context Windowが溢れて品質が崩壊する。「分ける」判断がエージェント設計の核心。

なぜ大事？

1人の営業が「テレアポ・提案書作成・契約書確認・入金管理」全部やったら？ → ミスが増え、どれも中途半端になる。AIも同じ。専門タスクごとに分けた方が、それぞれの品質が上がる。

分割の判断基準（3つのNG信号）

❌ ループ3回以上NG → 同じミスを3回繰り返したら、タスクが複雑すぎる合図
❌ 指示忘れが発生 → CWが圧迫されて、前の方の指示を忘れ始めている
❌ ハルシネーション発生 → 「嘘をつく」＝脳がパンクしている明確なサイン

→ この3つが出たら、サブエージェントに分割 or セッションを分ける
🏥 病院の例

1人の医師が内科・外科・眼科・歯科を全部やったら？
→ 専門性が薄まって全部中途半端になる
「専門医を分ける」のが、エージェント分割の発想。

🎬 映画制作の例

監督1人が脚本・撮影・照明・音声・編集を全部やったら？
→ 撮影中に脚本の直しが入って、照明の指示を忘れる
各部門にプロを置く = エージェント分割

うちの分割設計役割
commands/
オーケストレーター		ユーザーと対話。全体の指揮を取る（映画の「監督」）
agents/
サブエージェント		専門タスクを独立で実行（「照明」「音声」「編集」チーム）
親のCWを使わない = 本隊の頭脳を守る

🧲 ベクトル化 — 意味で検索する

「キーワード一致」から「意味が近い」への進化

ベクトル化とは、テキストを「数値の配列（ベクトル）」に変換すること。変換すると、テキスト同士の「意味の近さ」を数値で計算できるようになる。

なぜ大事？

従来の検索は「完全一致」。「犬」で検索しても「ワンちゃん」はヒットしない。ベクトル検索なら「犬」と「ワンちゃん」が意味的に近いことを理解して両方ヒットする。
これがないと、AIは「書いてあること」しか見つけられない。書いてあるのに見つけられない、は最悪。

📚 図書館の例

従来の検索（キーワード一致）
「犬の飼い方」で検索 → 「犬の飼い方」というタイトルの本だけヒット

ベクトル検索（意味検索）
「犬の飼い方」で検索 → 「はじめてのワンちゃん」「ペットとの暮らし」もヒット

🗂️ 社内ファイル検索の例

従来: 「売上報告」で検索 → ファイル名に「売上報告」がある資料だけ

ベクトル: 「売上報告」で検索 → 「Q3 Revenue Review」「第3四半期実績サマリー」も出てくる
言語の壁すら超える。

ベクトルの距離 = 意味の近さ

「膝がガクガクする」 ←→ 「膝が不安定」 = 距離0.15（ほぼ同じ意味）
「膝がガクガクする」 ←→ 「足が痛い」   = 距離0.45（関連あり）
「膝がガクガクする」 ←→ 「天気が良い」 = 距離0.92（無関係）

→ この「距離」を使って「似たものを探す」のがベクトル検索
「セマンティック検索ができるからLLMが重要。できないなら今までのPythonと同じ機能」 — 文太（AX社代表）

🔄 フィードバックループ — 自動で賢くなる仕組み

「やりっぱなし」を「やる→測る→直す→やる」に変える

フィードバックループとは、「実行結果を評価して、次の実行を改善する」自動サイクルのこと。AIが一発で終わるのではなく、自分で振り返って良くなっていく構造。

なぜ大事？

テスト勉強で「問題を解く → 丸つけしない → また同じ範囲を解く」を繰り返す人はいない。丸つけ（評価）して間違いを直す（改善）から成績が上がる。AIも同じ。実行→評価→改善のサイクルがないと、何回やっても同じ品質のまま。

🏋️ 筋トレの例

❌ 毎日なんとなく腕立て → 3ヶ月後も変化なし
✅ 記録つけて、重量を少しずつ上げて、フォーム確認 → 着実に成長
「記録 → 分析 → 調整」のサイクルが成長を生む。

📊 営業の例

❌ テレアポ100件 → 結果を見ない → また同じトークで100件
✅ テレアポ100件 → 「どのトークで何件取れたか」分析 → トーク改善
PDCAが回る営業チームは自然と数字が上がる。AIも同じ。

うちで目指す3つのループ

ループA: スキル自動改善
 スキル実行 → 品質スコア記録 → 3回分析 → 弱いステップを特定・修正

ループB: CRパフォーマンス
 広告CR生成 → 配信 → 実績データ取得 → 「勝ちパターン」を次回に反映

ループC: コンテキスト監視
 NGが3回続いたら → 自動で警告 → サブエージェントに分割提案

🪞 リフレクション — 振り返りの技術

「振り返って」だけでは曖昧。何をどう振り返るかまで指定する

リフレクションは「AIに自分の出力を振り返らせて修正させる」手法。ただし「品質を改善して」のような曖昧な指示では機能しない。

なぜ大事？

上司が新人に「もっとちゃんとやって」と言っても何も改善されない。「具体的にどこがダメで、どう直すか」まで示して初めて改善される。AIのリフレクションも同じ。

❌ NG — 曖昧なリフレクション指示

「品質を振り返って改善して」
「もっと良くして」
「前のと比較して」

✅ OK — 具体的なリフレクション指示

「各ステップの出力を比較し、以下の異常を検出して」
・トーンの急変（前ステップと感情スコア ±30%以上）
・情報の欠落（前ステップにあった必須項目が消えている）
・矛盾（前の結論と逆の主張が出現）
「異常があったステップのみ再実行」

📝 作文添削の例

❌「この作文をもっと良くしてください」
✅「この作文の①主語と述語のねじれ ②同じ接続詞の3回以上使用 ③段落冒頭の繰り返し表現をチェックして修正して」
チェック項目が具体的だから、改善が的確になる。

🏃 スポーツの例

❌ コーチ「もっと上手くなれ」
✅ コーチ「腕の振りが小さい。肘の角度を90度にして、腕を耳の横まで上げろ」
測定可能な改善ポイントを示すのがリフレクションの本質。

✍️ 指示の書き方 5要素

「言わなかったらAIが間違う方向に進むか？」がYESなら書く

AIへの指示は「過不足なく」書くのが理想。多すぎると読まれない。少なすぎると誤解される。判断基準は「これを書かなかったら、AIが間違った方向に進むか？」

5要素テンプレート

① 役割: あなたは[○○の専門家]です
② 目的: [○○]を達成してください
③ 背景: この案件は[○○]の文脈で[○○]なクライアント向けです
④ 制約: [○○]はしないでください / [○○]の範囲内で
⑤ 参考: [○○.md]を読んでから始めてください
🍕 ピザ注文の例

❌「ピザ頼んで」
→ サイズは？トッピングは？届け先は？

✅「マルゲリータのLサイズ、チーズ多め、○○町の自宅に30分以内で」
→ 迷わず注文できる。曖昧さを消すことで速度が上がる。

🏠 引越しの例

❌「来週引越し手伝って」
→ いつ？どこから？荷物の量は？

✅「来週土曜9時に渋谷のワンルームから新宿の2LDKに。段ボール20箱くらい。」
5W1Hが揃ってるから、すぐ動ける。

🛡️ 多層防御 — セキュリティ設計

「AIが間違えても被害が出ない」設計が多層防御

AIは間違える。これは前提。だから「間違えても大事故にならない仕組み」を複数の層で作る。AIに全てを任せず、要所で人間や機械的チェックを挟む。

なぜ大事？

飛行機が安全なのは、パイロットが優秀だからだけじゃない。自動操縦 + 副操縦士 + 管制塔 + 機体センサー…何重もの安全装置が「1つ壊れても大丈夫」を実現してる。AIエージェントも同じ。1層だけに頼ったらいつか事故る。

多層防御の設計パターン

第1層: スクリプト検証（機械的チェック）
 → 入力値のフォーマット確認、禁止ワードチェック等
 → LLMを使わない = 100%確実な防波堤

第2層: AI自己チェック（リフレクション）
 → 自分の出力に矛盾・異常がないか確認
 → 精度は高いが100%ではない

第3層: 人間の確認（最終ゲート）
 → 金額が大きい操作、公開する内容、不可逆な操作
 → 「送信ボタン」を押すのは人間

原則: バリデーション(検証)はスクリプトで。判断はAIで。最終承認は人間で。
🏦 銀行ATMの例

第1層: カード + 暗証番号（機械的認証）
第2層: 異常取引検知AI（パターン照合）
第3層: 高額送金は窓口で本人確認（人間チェック）
3層全部突破されないと被害が出ない設計。

📮 郵便の例

第1層: 郵便番号チェック（機械的 — 数字7桁か？）
第2層: 住所と郵便番号の整合性（ルールベース）
第3層: 配達員が「この住所おかしくない？」と判断（人間）
各層の役割が違う。全部同じチェックでは意味がない。

🗄️ ベクトルDBの活用 — データを資産にする

使い捨てにしない。蓄積して、未来の判断に活かす

ベクトル化したデータを「セッションごとに消す」のではなく、永続的なDBに蓄積する。すると「過去に成功したパターン」「失敗したパターン」が意味で検索できるようになる。

なぜ大事？

経験豊富なベテラン営業は「このタイプの顧客にはこの提案が刺さる」と分かっている。なぜか？ 過去の成功・失敗パターンが頭に蓄積されているから。ベクトルDBは「AIにベテランの経験値を与える」仕組み。

🍽️ レストランの例

❌ 毎日メニューを考える。前に何が人気だったか覚えていない
✅ 売上データを蓄積。「秋は和食の注文が1.5倍」と分かっている
データの蓄積 = 経験値の蓄積。過去のデータが未来の判断を助ける。

⚾ 野球の例

打者の過去データから「この投手の外角低めは打率.120」が分かれば、
外角低めを狙って投げる判断ができる。
ベクトルDBは「AIのスコアラー」。過去の記録から最適解を導く。

ベクトルの類似度意味判断
0.8以上ほぼ同じ差別化不足。角度を変える
0.7〜0.8同じカテゴリ良い方向性。微調整で使える
0.5〜0.7うっすら関連新しい切り口として検討価値あり
0.5未満無関係方向が違う
